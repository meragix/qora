Qora, `flutter_query`, and `cached_query` are all **server-state** libraries — purpose-built for data that lives on a server. Comparing them to Provider or BLoC would be like comparing a database client to a state container: different problems, different tools.

This page focuses on the meaningful differences between the three solutions in the same category.

For the full feature matrix, see [What is Qora? → How It Compares](./what-is-qora.md#how-it-compares).

---

## When to Choose

| If you need...                                           | Best fit              |
| --------------------------------------------------------- | --------------------- |
| A solid SWR cache with minimal setup                     | Any of the three      |
| Offline mutations queued and replayed on reconnect       | **Qora**              |
| Per-query NetworkMode (online / always / offlineFirst)   | **Qora**              |
| Thundering-herd control on reconnect                     | **Qora**              |
| Hooks API (useQuery, useMutation...)                     | **Qora** (qora_hooks) |
| First-class IDE and in-app DevTools                      | **Qora**              |
| Pure Dart core (no Flutter SDK dep)                      | **Qora**              |
| Existing community storage adapters (Hive, SharedPrefs)  | **cached_query**      |
| Smallest possible dependency footprint                   | **flutter_query**     |

## Where Qora Wins

### Offline-First Architecture

Qora is the only library in the Dart/Flutter ecosystem with a **first-class offline story**:

- **`NetworkMode` per query** — declare whether a query runs `online` only, `always` (local DB), or `offlineFirst` (serve cache, refresh in background when online).
- **Offline mutation queue** — writes submitted while offline are queued in order and replayed automatically on reconnect. `stopOnFirstError` handles sequentially-dependent operations.
- **`FetchStatus`** — a second-axis enum (`fetching / paused / idle`) emitted independently of `QoraState`, so the UI can distinguish "no network" from "loading".
- **`ReconnectStrategy`** — controls thundering-herd on reconnect: configurable max concurrent refetches and random jitter between batches.

```dart [user_screen.dart]
QoraBuilder<User>(
  queryKey: ['users', userId],
  queryFn: () => api.getUser(userId),
  options: const QoraOptions(networkMode: NetworkMode.offlineFirst),
  builder: (context, state, fetchStatus) {
    if (fetchStatus == FetchStatus.paused) {
      return OfflineBanner(staleData: state.dataOrNull);
    }
    return switch (state) {
      Success(:final data) => UserCard(data),
      Loading()            => const CircularProgressIndicator(),
      Failure(:final error)=> ErrorWidget('$error'),
      Initial()            => const SizedBox.shrink(),
    };
  },
)
```

### DevTools

Qora ships its own developer tooling — an in-app **overlay panel** and a full **IDE extension** — with zero configuration:

- Live cache inspector: key, state, data size, staleness
- Mutation timeline with before/after diff
- Network activity tab with fetch durations and error rates
- Performance tab with per-query stats
- Dependency graph

In 2026, DX is a first-class decision factor. No other Dart server-state library provides comparable tooling.

### Pure Dart Core

The `qora` package has no Flutter dependency. It runs in pure Dart CLI apps, Dart server-side code, and tests — without pulling in the Flutter SDK. `flutter_qora` is a thin opt-in layer of widget bindings on top.

### Advanced Invalidation

`invalidateQueries({required QueryFilter filter})` invalidates by state, last-used options, or any combination — not just by key prefix:

```dart [main.dart]
// Invalidate all online-mode queries that have not yet succeeded
client.invalidateQueries(
  filter: (key, state, opts) =>
      opts?.networkMode == NetworkMode.online && state is! Success,
);
```

## Where `cached_query` Has an Edge

`cached_query` has a more mature storage adapter ecosystem, with community-maintained adapters for Hive and SharedPreferences. Qora's `PersistQoraClient` covers the same use case with a clean `StorageAdapter` interface, but you write (or find) the adapter for your backend of choice.

## Complementarity with BLoC and Riverpod

Qora, BLoC, and Riverpod solve different problems and work well together:

| Layer                                                        | Right tool                   |
| ------------------------------------------------------------ | ---------------------------- |
| Data that lives on a server (queries, mutations, pagination) | **Qora**                     |
| Complex UI state machines (checkout flows, multi-step forms) | **BLoC**                     |
| Client state + DI (theme, auth session, service graph)       | **Riverpod** or **Provider** |

A production app commonly uses all three. See the integration guides if you are adding Qora to an existing BLoC or Riverpod codebase.

:::tip
Qora is designed for **server state**. It is perfectly complementary with BLoC or Riverpod, which excel at **client state** (navigation, UI toggles, form values, service wiring). Use each where it fits best.
:::
