Qora, `flutter_query`, and `cached_query` are server-state libraries built for data that resides on a remote server. Provider and BLoC target client state; comparing them to Qora conflates two distinct problem domains.

This page documents the meaningful differences between the three server-state solutions. For the full feature matrix, see [What is Qora? — Feature Comparison](./what-is-qora.md#feature-comparison).

---

## Selection Criteria

| If you need...                                          | Best fit              |
| ------------------------------------------------------- | --------------------- |
| A solid SWR cache with minimal setup                    | Any of the three      |
| Offline mutations queued and replayed on reconnect      | **Qora**              |
| Per-query NetworkMode (online / always / offlineFirst)  | **Qora**              |
| Thundering-herd control on reconnect                    | **Qora**              |
| Hooks API (useQuery, useMutation...)                    | **Qora** (qora_hooks) |
| First-class IDE and in-app DevTools                     | **Qora**              |
| Pure Dart core (no Flutter SDK dep)                     | **Qora**              |
| Existing community storage adapters (Hive, SharedPrefs) | **cached_query**      |
| Smallest possible dependency footprint                  | **flutter_query**     |

## Qora Differentiators

### Offline-First Architecture

Qora is the only library in the Dart/Flutter ecosystem with a complete offline architecture:

- **`NetworkMode` per query**: declare whether a query runs `online` only, `always` (local DB), or `offlineFirst` (serve cache, refresh in background when online).
- **Offline mutation queue**: writes submitted while offline are queued in order and replayed automatically on reconnect. `stopOnFirstError` handles sequentially-dependent operations.
- **`FetchStatus`**: a second-axis enum (`fetching / paused / idle`) emitted independently of `QoraState`, allowing the UI to distinguish "no network" from "loading".
- **`ReconnectStrategy`**: controls thundering-herd on reconnect with configurable max concurrent refetches and random jitter between batches.

```dart [user_screen.dart]
QoraBuilder<User>(
  queryKey: ['users', userId],
  fetcher: () => api.getUser(userId),
  options: const QoraOptions(networkMode: NetworkMode.offlineFirst),
  builder: (context, state, fetchStatus) {
    if (fetchStatus == FetchStatus.paused) {
      return OfflineBanner(staleData: state.dataOrNull);
    }
    return switch (state) {
      Success(:final data) => UserCard(data),
      Loading() => const CircularProgressIndicator(),
      Failure(:final error) => ErrorWidget('$error'),
      Initial() => const SizedBox.shrink(),
    };
  },
)
```

### DevTools

Qora ships dedicated developer tooling with zero configuration:

- Live cache inspector: key, state, data size, staleness
- Mutation timeline with before/after diff
- Network activity tab with fetch durations and error rates
- Performance tab with per-query statistics
- Dependency graph

No other Dart server-state library provides comparable tooling.

### Pure Dart Core

The `qora` package carries no Flutter dependency. It operates in CLI applications, Dart server-side code, and test suites without pulling in the Flutter SDK. `qora_flutter` is a thin opt-in widget binding layer on top.

### Advanced Invalidation

`invalidateQueries({required QueryFilter filter})` invalidates by state, last-used options, or any combination, not just by key prefix:

```dart [main.dart]
// Invalidate all online-mode queries that have not yet succeeded
client.invalidateQueries(
  filter: (key, state, opts) =>
      opts?.networkMode == NetworkMode.online && state is! Success,
);
```

## `cached_query`: Storage Adapter Ecosystem

`cached_query` has a more mature storage adapter ecosystem, with community-maintained adapters for Hive and SharedPreferences. Qora's `PersistQoraClient` addresses the same use case through a `StorageAdapter` interface, but adapters for specific backends must be sourced or implemented separately.

## Complementarity with BLoC and Riverpod

Qora, BLoC, and Riverpod address different concerns and are designed to be used together:

| Layer                                                        | Appropriate tool             |
| ------------------------------------------------------------ | ---------------------------- |
| Data that lives on a server (queries, mutations, pagination) | **Qora**                     |
| Complex UI state machines (checkout flows, multi-step forms) | **BLoC**                     |
| Client state and DI (theme, auth session, service graph)     | **Riverpod** or **Provider** |

A production application commonly uses all three in combination. Refer to the integration guides when adding Qora to an existing BLoC or Riverpod codebase.

:::tip
Qora manages server state. BLoC and Riverpod manage client state. The two domains are complementary; each library operates within its intended scope.
:::
