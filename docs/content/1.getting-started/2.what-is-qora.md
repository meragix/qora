**Qora** is a server state management library for Dart and Flutter, inspired by [TanStack Query](https://tanstack.com/query) and designed for the Dart/Flutter ecosystem.

## Problem Space

Server state is categorically different from local UI state. It resides on a remote system, has an independent lifecycle, and requires explicit synchronization with the client. Qora addresses this class of problem:

- **Cache** data with configurable TTL and serve it instantly on subsequent requests
- **Revalidate** stale data in the background without blocking the UI
- **Deduplicate** concurrent requests for the same key into a single network call
- **Retry** failed requests automatically with exponential backoff
- **Invalidate** targeted cache entries after mutations, triggering reactive refetches

## Core API

### One-shot fetch

```dart [user_repository.dart]
final user = await client.fetchQuery<User>(
  key: ['users', userId],
  fetcher: () => api.getUser(userId),
  options: const QoraOptions(staleTime: Duration(minutes: 5)),
);
```

Cache behavior: fresh data is returned immediately without a network call; stale data is returned immediately while a background refetch is dispatched.

### Reactive stream

```dart [user_repository.dart]
client.watchQuery<User>(
  key: ['users', userId],
  fetcher: () => api.getUser(userId),
).listen((state) {
  switch (state) {
    case Success(:final data):  render(data);
    case Failure(:final error): showError(error);
    default: {}
  }
});
```

### Invalidation after mutation

```dart [post_repository.dart]
await api.createPost(payload);

// Invalidate exact key
client.invalidate(['posts', postId]);

// Invalidate by predicate
client.invalidateWhere((key) => key.firstOrNull == 'posts');
```

### Optimistic updates

```dart [user_repository.dart]
final snapshot = client.getState<User>(['users', userId]);
client.setQueryData(['users', userId], user.copyWith(name: newName));

try {
  await api.updateUser(userId, newName);
} catch (_) {
  client.restoreQueryData(['users', userId], snapshot);
}
```

## Architecture

Qora is organized as two complementary packages:

### `qora`: Pure Dart core

No Flutter dependency. Operates in any Dart environment:

- CLI tools and scripts
- Backend Dart services
- Unit tests without the Flutter test runner
- Shared business logic packages

### `qora_flutter`: Flutter integration

A widget binding layer on top of `qora`:

- `QoraScope`: `InheritedWidget` that provides the client to the widget tree
- `QoraBuilder<T>`: fetches and reactively rebuilds on every state transition
- `QoraStateBuilder<T>`: observes state without triggering a fetch
- `QoraMutationBuilder<TData, TVariables, TContext>`: mutation state machine with lifecycle callbacks
- `InfiniteQueryBuilder<TData, TPageParam>`: paginated feeds with `fetchNextPage` / `fetchPreviousPage`
- `NetworkStatusIndicator` / `NetworkStatusBuilder`: offline banner and connectivity-aware widgets
- `FlutterLifecycleManager`: triggers refetch on app resume
- `FlutterConnectivityManager`: triggers refetch on network reconnect
- `context.qora`: shorthand extension for `QoraScope.of(context)`

### `qora_hooks`: flutter_hooks integration

Hooks-based API for projects using [flutter_hooks](https://pub.dev/packages/flutter_hooks):

- `useQuery<T>`: equivalent to `QoraBuilder` without the nested builder callback
- `useMutation<TData, TVariables>`: returns a `MutationHandle` with `mutate`, `isPending`, `isError`
- `useInfiniteQuery<TData, TPageParam>`: returns an `InfiniteQueryHandle` with `fetchNextPage`
- `useQueryClient`: returns the nearest `QoraClient` for imperative operations
- `useIsFetching` / `useIsMutating`: global `bool` indicators; rebuild only on zero-crossing

### DevTools

Debugging tooling for the Qora ecosystem:

- **IDE extension**: query inspector, mutation timeline, cache diff viewer inside the IDE
- **In-app overlay**: draggable panel displaying live cache state without leaving the simulator
- Both surfaces expose the same events: state transitions, fetch durations, optimistic diffs, and cancellations

## Design Principles

### Declarative state handling

The consumer declares what to render for each state variant. Qora manages fetch scheduling, cache resolution, and revalidation.

### Type-safe by construction

`QoraState<T>` is a sealed class. The Dart compiler enforces exhaustive pattern matching, eliminating unhandled state branches at compile time.

### Configurable defaults

Default configuration (`staleTime: Duration.zero`, `cacheTime: 5 minutes`, `retryCount: 3`, exponential backoff) is overridable globally via `QoraClientConfig` or per-query via `QoraOptions`.

## Feature Comparison

The table below compares Qora against libraries in the same category: `cached_query`, `flutter_query`, and Riverpod async providers. BLoC and Provider target client state (UI logic, forms, navigation) and are outside the scope of this comparison.

| Feature                                     | **Qora**            | cached\_query  | flutter\_query | Riverpod (async) |
| ------------------------------------------- | ------------------- | -------------- | -------------- | ---------------- |
| Stale-While-Revalidate                      | Built-in            | Built-in       | Built-in       | Manual           |
| Request deduplication                       | Built-in            | Built-in       | Built-in       | Manual           |
| Optimistic updates + rollback               | Built-in            | Manual         | Built-in       | Manual           |
| Automatic retry (exponential backoff)       | Built-in            | Built-in       | Built-in       | Plugin           |
| Infinite / paginated queries                | Built-in            | Built-in       | Built-in       | Manual           |
| Dependent queries (`dependsOn`)             | Built-in            | Manual         | ❌             | Built-in         |
| Cancel tokens                               | Built-in            | ❌             | ❌             | Manual           |
| Offline persistence (Hive, Isar...)         | Built-in            | Plugin         | ❌             | Manual           |
| **Offline mutation queue**                  | **Built-in**        | ❌             | ❌             | ❌               |
| **Network-aware refetch**                   | **Built-in**        | Basic          | ❌             | Manual           |
| **DevTools**                                | **Built-in**        | Plugin         | ❌             | Plugin           |
| Hooks API (useQuery, useMutation...)        | Plugin (qora_hooks) | ❌             | Built-in       | Built-in         |
| Pure Dart (no Flutter dependency)           | ✅                  | ❌             | ❌             | ❌               |

**Legend:** `Built-in`: zero configuration, ships with the package · `Plugin`: available via a separate package · `Manual`: achievable but requires custom implementation · `Partial`: exists but incomplete or undocumented · `❌`: not supported.

::callout
**Server state and client state are distinct problem domains.**

**Server state** is data that originates on a remote server. It has a lifecycle independent of the client: it is fetched, cached, becomes stale, and must be re-synchronized. The core engineering concerns are caching, deduplication, retries, and offline resilience.

**Client state** is data that exists exclusively in the application process: form values, selected tab, animation state, in-memory session tokens. It has no network lifecycle and does not go stale.

Production applications require both layers. Qora manages server state; BLoC, Riverpod, or Provider manage client state. The two domains are complementary and serve different purposes.
::

## Target Use Cases

Qora is suited to:

- **CRUD applications**: users, products, orders with reactive list and detail screens
- **Social feeds**: posts, likes, and comments synchronized across multiple screens
- **Dashboards**: polling for live data with background revalidation
- **E-commerce**: cart and checkout flows with optimistic UI
