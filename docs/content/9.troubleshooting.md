## Enable debug mode first

Before diagnosing anything, turn on `debugMode`. It prints `[Qora]` logs for every cache hit, miss, persist, hydrate skip, and serializer error — most issues become immediately obvious:

```dart [main.dart]
final client = QoraClient( // or PersistQoraClient
  config: const QoraClientConfig(debugMode: true),
);
```

## Persistence & Hydration

### Data is not persisted across restarts

**Symptom**: `PersistQoraClient` is used but the app always shows a spinner on cold start — no cached data.

**Cause**: No `QoraSerializer<T>` was registered for the type being fetched. Types without a serializer are cached in memory normally but never written to disk. No error is thrown.

```dart [main.dart]
// ✅ Register before calling hydrate()
client.registerSerializer<List<Post>>(
  QoraSerializer(
    toJson:   (posts) => posts.map((p) => p.toJson()).toList(),
    fromJson: (json)  => (json as List).map(Post.fromJson).toList(),
  ),
  name: 'List<Post>',
);
await client.hydrate();
```

With `debugMode: true`, an unregistered type produces: `[PersistQora] No serializer for type 'List<Post>' — skipping persist`.

---

### `hydrate()` runs but the cache is still empty

**Cause**: `registerSerializer` was called **after** `hydrate()`. During hydration, Qora looks up the serializer by the stored `typeName`. If no match is found, the entry is skipped.

```dart [main.dart]
// ❌ Wrong order
await client.hydrate();
client.registerSerializer<User>(...); // too late — User entries were skipped

// ✅ Correct order
client.registerSerializer<User>(...);
await client.hydrate();
runApp(...);
```

---

### Cache is warm but widgets still show a spinner on first frame

**Cause**: `hydrate()` was not awaited before `runApp`. The widget tree mounts before the cache is populated.

```dart [main.dart]
// ❌ hydrate() fires but isn't awaited
void main() {
  client.hydrate(); // no await
  runApp(QoraScope(client: client, child: const MyApp()));
}

// ✅ Await before runApp
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await storage.init();
  await client.hydrate();
  runApp(QoraScope(client: client, child: const MyApp()));
}
```

---

### Hydration breaks after a Flutter Web or `--obfuscate` build

**Cause**: Without an explicit `name:`, `registerSerializer` uses `T.toString()` as the type discriminator. Under obfuscation, `User.toString()` becomes `a` or `b`. The discriminator written in a debug build (`User`) doesn't match what's read in a release build (`a`), so all persisted entries are skipped.

```dart [main.dart]
// ❌ Unsafe — name becomes 'a' under obfuscation
client.registerSerializer<User>(userSerializer);

// ✅ Stable across all build modes
client.registerSerializer<User>(userSerializer, name: 'User');
client.registerSerializer<List<Post>>(postListSerializer, name: 'List<Post>');
```

The `name` must be **stable across app versions**. Changing it is equivalent to deleting all previously stored entries for that type.

---

### `fromJson` crashes on startup after a model change

**Cause**: A required field was added to the model between app versions. The old JSON on disk is missing the field, and `fromJson` throws.

`PersistQoraClient` catches every `fromJson` exception during `hydrate`, deletes the corrupt entry, and continues — the app never crashes. But the entry is lost and the user gets a network fetch on that run.

To prevent data loss, write a defensive `fromJson`:

```dart [models/user.dart]
factory User.fromJson(dynamic json) {
  final map = json as Map<String, dynamic>;
  return User(
    id:   map['id'] as int,
    name: map['name'] as String,
    // Added in v2 — default for entries persisted by v1
    role: map['role'] as String? ?? 'viewer',
  );
}
```

For breaking changes (type change, field removal with no safe default), bump the serializer `name` — e.g. `'User'` → `'User_v2'`. Old entries are silently skipped; the next successful fetch writes a fresh v2 entry.

## Query State

### Query is stuck in `Initial` — never fetches

**Cause A**: `enabled: false` is set on `QoraBuilder` or in `QoraOptions`. The widget observes state passively but does not trigger a fetch.

```dart
// Check this:
QoraBuilder<User>(
  enabled: someVariable, // if false → no fetch, state stays Initial
  ...
)
```

**Cause B**: `QoraStateBuilder` was used instead of `QoraBuilder`. `QoraStateBuilder` is observe-only — it never calls `fetchQuery`.

---

### Query is stuck in `Loading` indefinitely

**Cause A**: `dependsOn` is set and the dependency query has no data yet. `watchQuery` subscribes to the dependency and waits for it to reach `Success`. If the dependency itself has `enabled: false` or fails permanently, the dependent query never fires.

```dart
// Check both queries:
// 1. Is ['auth'] actually fetching and succeeding?
// 2. Is ['profile'] correctly declared as dependent?
client.watchQuery<Profile>(
  key: ['profile'],
  fetcher: () {
    final auth = client.getQueryData<Auth>(['auth'])!;
    return api.getProfile(auth.userId);
  },
  options: const QoraOptions(dependsOn: ['auth']),
);
```

**Cause B**: `fetchQuery` was used with `dependsOn` — it throws a `StateError` immediately if the dependency is not resolved. Use `watchQuery` for reactive dependent queries.

**Cause C**: `NetworkMode.online` (default) + no `ConnectivityManager` attached. Without a connectivity manager, the network mode is ignored and fetches always run. But if a `FlutterConnectivityManager` IS attached and the device is offline, the query pauses at `Loading(paused: true)` + `FetchStatus.paused`. This is correct behaviour — it will resume on reconnect.

---

### Query does not refetch when the app comes back to foreground

**Cause**: `FlutterLifecycleManager` was not passed to `QoraScope`.

```dart [main.dart]
// ❌ Missing lifecycleManager
QoraScope(client: client, child: const MyApp())

// ✅
QoraScope(
  client: client,
  lifecycleManager: FlutterLifecycleManager(qoraClient: client),
  child: const MyApp(),
)
```

Also check that `refetchOnWindowFocus: true` (the default) was not accidentally overridden globally in `QoraClientConfig.defaultOptions`.

---

### Query does not refetch when connectivity is restored

**Cause**: `FlutterConnectivityManager` was not passed to `QoraScope`.

```dart [main.dart]
// ✅
QoraScope(
  client: client,
  connectivityManager: FlutterConnectivityManager(),
  child: const MyApp(),
)
```

`FlutterConnectivityManager` requires `connectivity_plus` in `pubspec.yaml`. Without it the import will fail at compile time — so the more common mistake is simply forgetting to pass it to `QoraScope`.

## Cache & Keys

### Different widgets see different data for the "same" query

**Cause**: The query key is not structurally equal across call sites. Qora uses deep structural equality (`KeyEquality`) — `['users', 1]` and `['users', 1]` are the same key. But if one widget uses `['users', userId]` where `userId` is an `int` and another uses `['users', userId.toString()]` where `userId` is a `String`, they are **different keys** — two separate cache entries.

```dart
// ❌ Inconsistent key types
fetchQuery(key: ['users', 42], ...);    // int
watchQuery(key: ['users', '42'], ...);  // String — different key!

// ✅ Consistent
fetchQuery(key: ['users', 42], ...);
watchQuery(key: ['users', 42], ...);
```

---

### `QoraBuilder` re-subscribes on every rebuild (performance issue)

**Cause**: The `queryKey` is a non-`const` list literal created inline. `QoraBuilder.didUpdateWidget` compares the old and new key by reference — a new `List` instance is always unequal, so it re-subscribes and re-fetches every build.

```dart
// ❌ New List instance on every build
QoraBuilder<List<Post>>(
  queryKey: ['posts', page], // rebuilt each frame
  ...
)

// ✅ Stable reference — only changes when page changes
QoraBuilder<List<Post>>(
  queryKey: const ['posts'],         // for static keys
  // or
  queryKey: ['posts', page],         // fine IF page is stable (not rebuilt every frame)
  ...
)
```

If the key contains variables, make sure those variables only change when the data actually needs to change — not on every `setState`.

---

### `dataOrNull` returns `null` even after a successful fetch

**Cause**: The type parameter on `QoraBuilder<T>` doesn't match what was fetched. The state stream is typed — a `Success<User>` emitted for key `['users', 1]` will not be received by `QoraBuilder<Profile>` on the same key.

```dart
// ❌ Type mismatch — state is Success<User> but builder expects Profile
QoraBuilder<Profile>(queryKey: ['users', userId], ...)

// ✅
QoraBuilder<User>(queryKey: ['users', userId], ...)
```

## Optimistic Updates

### Rollback doesn't work — UI stays in the optimistic state after failure

**Cause**: The snapshot was taken after `setQueryData` instead of before.

```dart
// ❌ Snapshot is taken after the optimistic write — it captures the wrong state
client.setQueryData<User>(key, optimisticUser);
final snapshot = client.getState<User>(key); // already optimistic!

// ✅ Always snapshot first
final snapshot = client.getState<User>(key); // save the real state
client.setQueryData<User>(key, optimisticUser);
try {
  await api.updateUser(...);
} catch (_) {
  client.restoreQueryData<User>(key, snapshot); // restores exact pre-mutation state
}
```

---

### Using `setQueryData` for rollback instead of `restoreQueryData`

**Cause**: `setQueryData` always produces a `Success` state with `DateTime.now()` as `updatedAt`. If the original state was `Initial` or `Loading`, `setQueryData` produces the wrong state. If the original `updatedAt` was recent (fresh data), `setQueryData` produces a stale timestamp.

`restoreQueryData` accepts a full `QoraState<T>` snapshot — it restores the exact state type, `updatedAt`, and `previousData`.

```dart
// ❌ Always produces Success — loses original state type and timestamp
client.setQueryData<User>(key, previousUser);

// ✅ Restores exact pre-mutation state
client.restoreQueryData<User>(key, snapshot);
```

## Flutter Integration

### `context.qora` throws `FlutterError: QoraScope not found`

**Cause**: No `QoraScope` ancestor exists above the widget that calls `context.qora` (or `QoraScope.of(context)`).

```dart [main.dart]
// ✅ QoraScope must be above the widget tree
runApp(
  QoraScope(
    client: client,
    child: const MyApp(), // context.qora works anywhere inside MyApp
  ),
);
```

Common trap: calling `context.qora` inside a callback that fires after the widget is disposed, or inside an `initState` of a widget that sits above `QoraScope` in the tree.

## Offline Mutation Queue

### Mutations are not replayed after reconnect

**Cause A**: `FlutterConnectivityManager` not passed to `QoraScope`. Without it, Qora never receives the reconnect signal.

**Cause B**: `offlineQueue: true` was not set on `MutationOptions`. The offline queue is opt-in per mutation.

```dart
QoraMutationBuilder<void, Todo, void>(
  mutationFn: api.createTodo,
  options: MutationOptions(
    offlineQueue: true, // ← required to enqueue when offline
  ),
  ...
)
```

---

### Queue replay stops after the first mutation — other mutations never run

**Cause**: `stopOnFirstError: true` is configured and the first mutation in the queue returned a **server error** (4xx/5xx) — not a network error. `stopOnFirstError` was designed for ordering dependencies, but a permanent 4xx will keep blocking the queue on every reconnect.

Options:
1. Leave `stopOnFirstError: false` (default) if your mutations are independent.
2. If ordering is required, inspect the error type in `onError` and remove irrecoverable mutations from the queue manually before they block.

```dart
final client = QoraClient(
  // Only use stopOnFirstError when mutations have strict ordering dependencies
  offlineMutationQueue: OfflineMutationQueue(stopOnFirstError: true),
);
```

## Infinite Queries

### `fetchNextPage()` is called but no new data loads

**Cause**: `getNextPageParam` returned `null` for the last page, setting `hasNextPage` to `false`. Qora correctly does nothing — it considers the feed complete. Check your pagination logic:

```dart
// Integer pages — common mistake: returns null too early
getNextPageParam: (lastPage, allPages) =>
    lastPage.items.isEmpty ? null : allPages.length + 1,
//  ↑ correct: null when truly empty

// Cursor-based — common mistake: uses a wrong field
getNextPageParam: (lastPage, _) => lastPage.nextCursor,
// ↑ if nextCursor is an empty string "" instead of null, hasNextPage stays true forever
// Fix: lastPage.nextCursor?.isNotEmpty == true ? lastPage.nextCursor : null
```

---

### Feed shows duplicate items after `refetch()`

**Cause**: Integer page-number pagination combined with server-side insertions. When new posts are added between `page=1` and `page=2`, re-fetching `page=1` returns different items than the first load, causing overlap.

**Fix**: Use cursor-based pagination. Cursors anchor to a stable server-side position and are unaffected by insertions.

```dart
// ❌ Integer pages — unstable with server-side insertions
InfiniteQueryOptions(
  initialPageParam: 1,
  getNextPageParam: (_, pages) => pages.length + 1,
)

// ✅ Cursor — stable across insertions
InfiniteQueryOptions<FeedPage, String?>(
  initialPageParam: null,
  getNextPageParam: (last, _) => last.nextCursor,
)
```
