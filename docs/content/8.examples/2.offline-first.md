## Objective

**Context**: Mobile connectivity is intermittent by design. Mutations lost during offline periods create data divergence and operational risk. An offline-first system guarantees eventual delivery in submission order without duplicate writes.

This example demonstrates how to achieve this using Qora's persistence and network-aware layers, without custom synchronization logic.

---

## Key Features Demonstrated

- **`PersistQoraClient`**: write-through cache that survives app restarts via Hive
- **`hydrate()`**: warms the cache from disk before the first frame renders; returning users never see a spinner
- **`NetworkMode.offlineFirst`**: serves stale data instantly and schedules background revalidation on reconnect
- **`OfflineMutationQueue` with `stopOnFirstError: true`**: FIFO replay that preserves create→update ordering
- **`optimisticResponse` + `isOptimistic` flag**: UI reflects writes instantly with a clear "pending sync" indicator
- **`FetchStatus.paused`**: surfaces the offline state to the UI without any custom state management
- **`NetworkStatusIndicator`**: automatic offline banner with zero widget boilerplate
- **`ReconnectStrategy`**: controlled concurrency and jitter to prevent thundering-herd on reconnect

## Architecture Highlights

The key architectural decision is that `PersistQoraClient` is a **drop-in replacement** for `QoraClient`: it extends the base class and every existing API (`fetchQuery`, `watchQuery`, `invalidate`, …) works unchanged. The rest of the app is unaware of persistence.

```text
main.dart
├── HiveStorageAdapter.init()          ← open the Hive box
├── PersistQoraClient(storage, ...)    ← drop-in replacement for QoraClient
├── registerAllSerializers(client)     ← register Todo + List<Todo> serializers
├── client.hydrate()                   ← inject disk data into cache before runApp
└── QoraScope(
      client: client,
      lifecycleManager: FlutterLifecycleManager(...),    ← refetch on app resume
      connectivityManager: FlutterConnectivityManager(), ← trigger reconnect replay
      child: NetworkStatusIndicator(child: MyApp()),     ← offline banner, zero config
    )
```

Serializers are collected in a single `registerAllSerializers()` function: this becomes the authoritative list of every type written to disk, useful for auditing during model migrations.

```dart [lib/core/persistence/serializers.dart]
void registerAllSerializers(PersistQoraClient client) {
  client.registerSerializer<Todo>(
    QoraSerializer(toJson: (t) => t.toJson(), fromJson: Todo.fromJson),
    name: 'Todo', // explicit name: survives Flutter Web obfuscation
  );
  client.registerSerializer<List<Todo>>(
    QoraSerializer(
      toJson: (list) => list.map((t) => t.toJson()).toList(),
      fromJson: (json) => (json as List).map(Todo.fromJson).toList(),
    ),
    name: 'List<Todo>',
  );
}
```

The `OfflineMutationQueue` is configured with `stopOnFirstError: true` at the client level, because todo mutations have ordering dependencies (a todo must exist before it can be completed or deleted).

```dart [lib/main.dart]
final client = PersistQoraClient(
  storage: HiveStorageAdapter(boxName: 'qora_cache'),
  persistDuration: const Duration(days: 30),
  offlineMutationQueue: OfflineMutationQueue(stopOnFirstError: true),
  config: const QoraClientConfig(
    defaultOptions: QoraOptions(
      staleTime: Duration(minutes: 5),
      networkMode: NetworkMode.offlineFirst,
    ),
    reconnectStrategy: ReconnectStrategy(
      maxConcurrent: 3,
      jitter: Duration(milliseconds: 150),
    ),
    debugMode: kDebugMode,
  ),
);
```

## Implementation Details

### The ordering problem: why `stopOnFirstError` matters

Consider this offline session:

```text
1. User creates Todo("Buy milk")   → enqueued as PendingMutation #1
2. User marks it complete          → enqueued as PendingMutation #2 (references the todo's ID)
3. Connectivity returns
4. Queue replays: #1 fires → server creates todo → returns { id: "srv-42" }
5. Queue replays: #2 fires → PUT /todos/srv-42/complete → ✅
```

If `stopOnFirstError` were `false` and mutation #1 failed (e.g. server returned 422), mutation #2 would still fire: referencing an ID that does not exist on the server. With `stopOnFirstError: true`, mutation #2 is re-enqueued at the front and retried on the next reconnect, after the root cause of #1 is resolved.

### The `isOptimistic` feedback loop

When a user creates a todo offline, the mutation emits `MutationSuccess(isOptimistic: true)`: the item appears in the list immediately. The `isOptimistic` flag drives a "pending sync" visual that provides honest, non-alarming feedback:

```dart [lib/ui/todo_list.dart]
QoraMutationBuilder<Todo, CreateTodoInput, void>(
  mutator: (input) => api.createTodo(input),
  options: MutationOptions(
    offlineQueue: true,
    // Provide a local optimistic object: appears in UI immediately
    optimisticResponse: (input) => Todo(
      id: 'temp-${DateTime.now().millisecondsSinceEpoch}',
      title: input.title,
      completed: false,
      isPending: true,
    ),
    onSuccess: (_, __, ___) async {
      // Runs for REAL success only (isOptimistic: false)
      // The server response replaces the temp ID: invalidate to re-sort
      context.qora.invalidate(const ['todos']);
    },
  ),
  builder: (context, mutationState, mutate) {
    final isQueued = switch (mutationState) {
      MutationSuccess(:final isOptimistic) => isOptimistic,
      _ => false,
    };

    return FloatingActionButton.extended(
      onPressed: mutationState.isPending ? null : () => mutate(input),
      icon: isQueued
          ? const Icon(Icons.schedule) // clock icon when queued offline
          : const Icon(Icons.add),
      label: Text(isQueued ? 'Queued: syncing soon' : 'Add Todo'),
    );
  },
)
```

And in each `TodoTile`, the optimistic item is rendered with a subtle pending style:

```dart [lib/ui/todo_tile.dart]
ListTile(
  title: Text(
    todo.title,
    style: TextStyle(
      color: todo.isPending ? Colors.grey.shade500 : null,
      fontStyle: todo.isPending ? FontStyle.italic : null,
    ),
  ),
  leading: todo.isPending
      ? const SizedBox(
          width: 20,
          height: 20,
          child: CircularProgressIndicator(strokeWidth: 2),
        )
      : Checkbox(value: todo.completed, onChanged: (_) => _complete(todo)),
  trailing: todo.isPending
      ? Tooltip(
          message: 'Will sync when online',
          child: Icon(Icons.cloud_off, size: 16, color: Colors.grey.shade400),
        )
      : null,
)
```

### Cold-start: zero spinner for returning users

`hydrate()` is awaited before `runApp`. The restored `Success` state carries the original `persistedAt` timestamp. Because `staleTime` is 5 minutes, a user who opens the app within 5 minutes of their last session sees their todo list **on the first frame**: no network call, no spinner. After 5 minutes, the cached list is shown immediately while Qora revalidates in the background (SWR):

```dart [lib/main.dart]
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();

  final storage = HiveStorageAdapter(boxName: 'qora_cache');
  await storage.init();

  final client = PersistQoraClient(storage: storage, /* ... */);
  registerAllSerializers(client);
  await client.hydrate(); // ← warm cache before first frame

  runApp(QoraScope(client: client, child: const MyApp()));
}
```

### Handling the `FetchStatus.paused` state

`QoraBuilder` exposes `fetchStatus` as its third builder argument. When the device is offline and a query has no cached data, the state is `Loading(paused: true)` + `FetchStatus.paused`. This is distinct from `Loading` + `FetchStatus.fetching` (actively loading) and must be rendered differently:

```dart [lib/ui/todo_screen.dart]
QoraBuilder<List<Todo>>(
  queryKey: const ['todos'],
  fetcher: api.getTodos,
  builder: (context, state, fetchStatus) {
    // Special case: offline + no cache at all
    if (state is Initial && fetchStatus == FetchStatus.paused) {
      return const Center(
        child: Column(
          mainAxisAlignment: MainAxisAlignment.center,
          children: [
            Icon(Icons.wifi_off, size: 48, color: Colors.grey),
            SizedBox(height: 16),
            Text('No connection: connect to load your todos'),
          ],
        ),
      );
    }

    return switch (state) {
      Initial() || Loading(previousData: null) => const CircularProgressIndicator(),
      Failure(:final error, previousData: null) => ErrorScreen(error),
      // Data available: Success, or Loading/Failure with cached previousData
      _ => TodoList(
          todos: state.dataOrNull ?? [],
          isRefreshing: fetchStatus == FetchStatus.fetching,
          isOffline: fetchStatus == FetchStatus.paused,
        ),
    };
  },
)
```
