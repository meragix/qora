## Objective

**Context**: An infinite scroll feed sounds simple: load more data as the user scrolls. The production constraints are strict: after 30 minutes of scrolling, your `ListView` holds 2,000 widgets, memory spikes, and the OS terminates your app. Alternatively, if you use integer page numbers and a post is inserted mid-scroll, every subsequent page is offset by one and users see duplicates.

This example demonstrates two core controls. Cursor-based pagination eliminates offset drift. Windowed paging with `maxPages` enforces a bounded in-memory footprint while allowing backward page recovery on demand.

---

## Key Features Demonstrated

- **`InfiniteQueryObserver` / `InfiniteQueryBuilder`**: the core engine for accumulating pages
- **Cursor-based pagination**: stable results regardless of server-side insertions
- **`maxPages` windowed memory management**: caps the in-memory page window; drops oldest/newest when exceeded
- **`isFetchingNextPage` + `isFetchingPreviousPage`**: drives the scroll-end loader without hiding existing content
- **`InfiniteData.flatten()`**: converts the page matrix into a flat item list for `SliverList`
- **Optimistic post creation**: `setInfiniteQueryData` prepends a new post instantly; rollback on failure
- **`refetch()`**: pull-to-refresh that re-fetches all loaded pages while preserving the page count
- **`InfiniteFailure` with `previousData`**: load-more errors show a retry button without clearing the feed

## Architecture Highlights

The UI layer is built on a `CustomScrollView` with two slivers: a `CupertinoSliverRefreshControl` for pull-to-refresh and a `SliverList.builder` for the feed items. The `InfiniteQueryBuilder` wraps the scroll view and owns the observer lifecycle.

The `InfiniteQueryObserver` is configured with a `String?` cursor parameter. `null` means "start from the beginning": the server returns the most recent posts and a `nextCursor` pointing to older content.

```dart [lib/features/feed/data/feed_observer.dart]
final observer = InfiniteQueryObserver<FeedPage, String?>(
  client: client,
  key: const ['feed'],
  fetcher: (cursor) => api.getFeed(cursor: cursor, limit: 20),
  options: InfiniteQueryOptions(
    initialPageParam: null,                            // null = latest posts
    getNextPageParam:     (last, _) => last.nextCursor,     // older posts
    getPreviousPageParam: (first, _) => first.previousCursor, // newer posts
    maxPages: 8,   // в‰€ 160 posts in memory: well within Flutter's budget
    baseOptions: const QoraOptions(
      staleTime: Duration(minutes: 2),
      retryCount: 3,
    ),
  ),
);
```

The `maxPages: 8` constraint is the key production decision. At 20 posts per page, the widget tree never exceeds ~160 items: Flutter's `SliverList.builder` renders only visible items anyway, but this bounds the `InfiniteData` object itself.

## Implementation Details

### Windowed paging: scrolling back in time

When the user scrolls down far enough to load page 9, `maxPages: 8` causes page 1 (the newest posts) to be **dropped** from `InfiniteData`. The state's `hasPreviousPage` becomes `true`. If the user scrolls back up to the top, the `SliverList` detects this and calls `observer.fetchPreviousPage()`, which re-fetches the dropped page using `getPreviousPageParam`.

This entire mechanism: drop, detect, re-fetch: is why **cursor-based pagination is mandatory** with `maxPages`. Integer page numbers would break here: if 3 new posts were inserted while the user was scrolling, re-fetching "page 1" with `?page=1&limit=20` would return a different 20 posts than the first load. Cursors anchor to a specific server-side position.

```dart [lib/features/feed/ui/feed_screen.dart]
void _onScroll() {
  final pos = _scrollController.position;

  // Trigger next page when within 300px of the bottom
  if (pos.extentAfter < 300 &&
      _latestState case InfiniteSuccess(:final hasNextPage, :final isFetchingNextPage)
          when hasNextPage && !isFetchingNextPage) {
    observer.fetchNextPage();
  }

  // Trigger previous page when within 300px of the top
  // (only relevant after maxPages eviction has dropped page 1)
  if (pos.extentBefore < 300 &&
      _latestState case InfiniteSuccess(:final hasPreviousPage, :final isFetchingPreviousPage)
          when hasPreviousPage && !isFetchingPreviousPage) {
    observer.fetchPreviousPage();
  }
}
```

### Optimistic post creation with `setInfiniteQueryData`

When the user creates a new post, it should appear at the top of the feed instantly: before the server responds. Qora exposes `setInfiniteQueryData` to write directly into the infinite cache:

```dart [lib/features/feed/ui/compose_sheet.dart]
Future<void> _publish(String content) async {
  final client = context.qora;
  const feedKey = ['feed'];

  // 1. Snapshot for rollback
  final snapshot = client.getInfiniteQueryData<FeedPage, String?>(feedKey);

  // 2. Optimistic prepend to first page
  if (snapshot != null) {
    client.setInfiniteQueryData<FeedPage, String?>(
      feedKey,
      InfiniteData(
        pages: [
          FeedPage(
            posts: [
              Post.optimistic(
                id: 'temp-${DateTime.now().millisecondsSinceEpoch}',
                content: content,
                author: currentUser,
              ),
              ...snapshot.pages.first.posts,
            ],
            nextCursor: snapshot.pages.first.nextCursor,
            previousCursor: null, // optimistic post has no stable cursor yet
          ),
          ...snapshot.pages.skip(1),
        ],
        pageParams: snapshot.pageParams,
      ),
    );
  }

  try {
    // 3. Fire the real mutation
    await api.createPost(content);

    // 4. Invalidate: replace the optimistic item with the server response
    //    refetch() re-fetches all current pages preserving the page count
    await observer.refetch();
  } catch (error) {
    // 5. Rollback: restore the exact pre-mutation snapshot
    if (snapshot != null) {
      client.setInfiniteQueryData(feedKey, snapshot);
    }
    _showError(error);
  }
}
```

The critical detail here is **step 4**: calling `observer.refetch()` rather than `invalidateInfiniteQuery`. `refetch()` re-fetches all currently loaded pages (8 pages Г— 20 items = 160 posts re-validated in sequence), which replaces the temp-ID optimistic post with the real server object. `invalidateInfiniteQuery` would reset the state to `InfiniteInitial` and restart from page 1, losing the user's scroll position.

### `InfiniteFailure` with graceful degradation

Load-more errors must never erase the feed the user already loaded. `InfiniteFailure` preserves `previousData`: the last successful `InfiniteData`: for exactly this purpose:

```dart [lib/features/feed/ui/feed_screen.dart]
case InfiniteFailure(:final error, :final previousData):
  if (previousData != null) {
    final posts = previousData.flatten((p) => p.posts);
    return SliverList.builder(
      itemCount: posts.length + 1, // +1 for the error footer
      itemBuilder: (context, i) {
        if (i < posts.length) return PostTile(post: posts[i]);
        // Error footer at the very end
        return _LoadMoreErrorBanner(
          error: error,
          onRetry: observer.fetchNextPage,
        );
      },
    );
  }
  // No data at all: full error screen
  return SliverFillRemaining(child: ErrorScreen(error));
```

### The `flatten` call

`InfiniteData<FeedPage, String?>` holds a `List<FeedPage>`: one item per loaded page. `flatten` extracts the inner list from each page and concatenates them:

```dart [lib/features/feed/ui/feed_screen.dart]
case InfiniteSuccess(:final data, :final hasNextPage, :final isFetchingNextPage):
  final posts = data.flatten((page) => page.posts); // List<Post>

  return SliverList.builder(
    itemCount: posts.length + 1,
    itemBuilder: (context, index) {
      if (index < posts.length) return PostTile(post: posts[index]);

      // Footer: spinner, "load more" trigger, or end-of-feed message
      if (isFetchingNextPage) return const _PageLoader();
      if (hasNextPage) return _LoadMoreButton(onTap: observer.fetchNextPage);
      return const _EndOfFeedBanner();
    },
  );
```

## Performance Notes

| Concern | Solution |
|---|---|
| Memory bloat from long sessions | `maxPages: 8` caps the page window; evicted pages are re-fetched on demand |
| Scroll jank from large lists | `SliverList.builder` renders only visible items; `InfiniteData` object itself is bounded |
| Cursor drift from server inserts | Cursor-based pagination anchors each page to a stable server-side position |
| Thundering-herd on `refetch()` | Pages are re-fetched **sequentially**, not concurrently |
| Stale feed on return to tab | `staleTime: 2 minutes` triggers a background `refetch()` on tab re-focus via `FlutterLifecycleManager` |
