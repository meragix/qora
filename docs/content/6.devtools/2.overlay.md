## Installation

Add `qora_devtools_overlay` to `dev_dependencies`, not `dependencies`. This keeps the package out of production bundles.

```yaml [pubspec.yaml]
dev_dependencies:
  qora_devtools_overlay: ^0.1.0
```

---

## Setup

Create an `OverlayTracker`, pass it to `QoraClient`, `QoraInspector`, and — to enable the inspector action buttons — pass the client reference to `QoraInspector` as well.

```dart [main.dart]
import 'package:flutter/foundation.dart';
import 'package:qora/qora.dart';
import 'package:qora_devtools_overlay/qora_devtools_overlay.dart';

void main() {
  final tracker = OverlayTracker();

  final client = QoraClient(
    // null in release → QoraClient uses NoOpTracker, no events emitted.
    tracker: kDebugMode ? tracker : null,
  );

  runApp(
    QoraInspector(
      tracker: tracker,
      // Required for action buttons (Refetch, Invalidate, Remove, etc.)
      // to operate on the live cache. Omitting this hides the ACTIONS section.
      client: kDebugMode ? client : null,
      child: MyApp(client: client),
    ),
  );
}
```

In debug mode, a floating **Q** button is rendered in the bottom-right corner of the app.

::callout{type="warning"}
The `client` parameter is optional by design. `QoraInspector` functions as a read-only observer without it. When omitted, the **ACTIONS** section in the query inspector is not rendered.
::

::callout{type="info"}
`QoraInspector` renders nothing in release builds. The guard is applied at widget level, so the overlay subtree is not constructed outside `kDebugMode`.
::

## The panel

Tap the **Q** button to open the panel. It slides up from the bottom of the screen, covering 60% of the height.

The panel has two main tabs.

### Queries

A two-column layout for inspecting cached queries.

#### Column 1: Query list

Lists every query currently tracked by the `QoraClient`. Each row displays the query key, status badge, and last updated time.

Tap any row to open its detail in column 2.

#### Column 2: Query inspector

Shows the full detail of the selected query. Three sections are displayed.

**Query Key**: the normalised key in readable form, for example `[ "users", "42" ]`.

**Actions** (requires `client:` to be set on `QoraInspector`): direct operations on the live cache.

| Action         | Effect                                                             |
| -------------- | ------------------------------------------------------------------ |
| Refetch        | Marks the query stale and triggers an immediate background refetch |
| Invalidate     | Marks the query stale; active observers refetch on next access     |
| Remove         | Removes the cache entry entirely                                   |
| Mark Stale     | Sets the stale flag without triggering a refetch                   |
| Simulate Error | Forces the query into `Failure` state for UI testing               |

**Metadata**: `Updated At`, `Stale At`, `Cache Time`, `Observer Count`, `Fetch Duration`.

### Mutations

A three-column layout that mirrors the structure of the IDE extension.

#### Column 1: Mutations list

Shows all mutations with their key, status, and elapsed time. Each row displays:

- A status indicator (green for success, blue for pending, orange for error)
- The query key as a readable breadcrumb, for example `user › 42 › update`
- An `⚡ Optimistic` badge when an optimistic update was applied
- A `Retries: N` badge when the mutation has been retried

Tap any row to open its detail in column 2.

#### Column 2: Mutation inspector

Shows the full detail of the selected mutation:

- **Status**: Current status badge (`error`, `success`, `pending`)
- **Actions**: `Retry` button to re-trigger the mutation immediately
- **Variables**: The input passed to the mutation, expandable
- **Error**: The error object if the mutation failed, expandable
- **Rollback context**: The snapshot saved before the optimistic update was applied
- **Metadata**: Created at, submitted at, updated at, retry count

#### Column 3: Secondary tabs

Three tabs give additional context for the selected mutation.

**Timeline**: Chronological log of session events: `OptimisticUpdate`, `MutationStarted`, `QueryCreated`, `FetchStarted`, `FetchError`, `MutationError`, `MutationSuccess`. Each event includes type, query key, and millisecond timestamp. Use the toolbar to filter by key or event type, pause the stream, or clear the log.

**Widget Tree**: Planned.

**Data Diff**: a side-by-side comparison of the mutation's input and output. The **BEFORE** column shows the variables submitted to the mutator. The **AFTER** column shows the result returned from the server (green header on success, red on error, grey while pending). Both columns render the payload as an interactive JSON tree you can expand and collapse inline.

## Responsive behavior

The overlay adapts to the available screen size.

- **≥ 600 px** (tablet, desktop): Three-column layout with all panels visible.
- **< 600 px** (phone): Single-column navigation with forward navigation into inspector and timeline views.

## Production safety

Three independent layers ensure the overlay never reaches production:

### 1. `dev_dependencies`

The package is physically absent from your release bundle because it lives in `dev_dependencies`.

### 2. Widget guard

```dart
@override
Widget build(BuildContext context) {
  if (!kDebugMode) return child; // nothing is built in release
  return _buildOverlay();
}
```

### 3. NoOpTracker in release

`QoraClient` is constructed with `tracker: kDebugMode ? tracker : null`. In release mode, `null` activates `NoOpTracker`. Its methods are no-ops and no tooling events are emitted.

Result: no production runtime impact from the overlay tracker path.
