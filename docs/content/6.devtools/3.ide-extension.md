title: IDE Extension

---

The Qora IDE extension adds a dedicated tab to Flutter DevTools. It is accessible via **Flutter: Open DevTools** in VS Code or **Tools > Flutter > Open DevTools** in IntelliJ.

The extension communicates with the running app over the Dart VM Service. It is compatible with simulators, emulators, and physical devices connected via USB.

---

## Installation

Add `qora_devtools_extension` to `dependencies`, not `dev_dependencies`. The package registers VM Service extensions at startup. Tree shaking removes the relevant code from release builds automatically.

```yaml [pubspec.yaml]
dependencies:
  qora: ^0.4.0
  qora_devtools_extension: ^0.3.0
```

Call `QoraDevtools.setup` once in `main`, before `runApp`:

```dart [main.dart]
import 'package:flutter/foundation.dart';
import 'package:qora_devtools_extension/qora_devtools_extension.dart';

void main() {
  final client = QoraClient();

  if (kDebugMode) QoraDevtools.setup(client);

  runApp(MyApp(client: client));
}
```

`QoraDevtools.setup` performs three operations in sequence: it initializes a `LazyPayloadManager`, constructs a `VmTracker` backed by that manager, and registers all `ext.qora.*` VM Service extension handlers.

### With the in-app overlay

When the [in-app overlay](/devtools/overlay) is also used, pass the `OverlayTracker` via `additionalTrackers`. Both surfaces receive the same events from a shared `MultiTracker`:

```dart [main.dart]
import 'package:flutter/foundation.dart';
import 'package:qora_devtools_extension/qora_devtools_extension.dart';
import 'package:qora_devtools_overlay/qora_devtools_overlay.dart';

void main() {
  final overlay = OverlayTracker();
  final client  = QoraClient();

  if (kDebugMode) {
    QoraDevtools.setup(client, additionalTrackers: [overlay]);
  }

  runApp(
    QoraInspector(
      tracker: overlay,
      client: kDebugMode ? client : null,
      child: MyApp(client: client),
    ),
  );
}
```

::callout{type="info"}
Each tracker passed via `additionalTrackers` is wrapped in a `MultiTracker`. Trackers receive events in declaration order.
::

### Manual wiring

When direct control over individual components is required (for example, to share a single `LazyPayloadManager` across multiple clients or to set a custom ring-buffer size), the components can be wired individually:

```dart [main.dart]
import 'package:qora_devtools_extension/qora_devtools_extension.dart';

void main() {
  final lazy    = LazyPayloadManager();
  final tracker = VmTracker(lazyPayloadManager: lazy, maxBuffer: 1000);
  final client  = QoraClient();

  client.setTracker(tracker);

  ExtensionRegistrar(
    handlers: ExtensionHandlers(
      gateway: QoraClientTrackingGateway(client),
      lazyPayloadManager: lazy,
    ),
  ).registerAll();

  runApp(MyApp(client: client));
}
```

> The `LazyPayloadManager` instance passed to `VmTracker` must be identical to the one passed to `ExtensionHandlers`. The two components share it to coordinate chunked payload storage and retrieval.

## Opening the extension

1. Run the app in debug mode.
2. Open Flutter DevTools.
   - **VS Code**: run **Flutter: Open DevTools** from the command palette.
   - **IntelliJ**: **Tools > Flutter > Open DevTools**.
3. Select the **Qora** tab in the DevTools tab bar.

The extension connects to the running isolate without requiring URL or port configuration.

## The panel

The extension panel is organized into six tabs.

### Queries

The Queries tab displays every query tracked by `QoraClient`, updated as events arrive. Each row shows:

- The query key rendered as a breadcrumb sequence (`user › 42 › profile`).
- A status badge (`success`, `loading`, `error`, `stale`, `idle`).
- Approximate payload size and last-updated timestamp.
- An expandable inline JSON tree. Large payloads are fetched on demand via the chunked transfer protocol.

The **Refetch** and **Invalidate** buttons on each row trigger the corresponding action on the device. The filter bar searches by key substring. The **Refresh** button forces a full cache snapshot sync.

### Mutations

The Mutations tab uses a three-column layout.

#### Column 1: Mutations list

Each row shows:

- A status indicator (green / blue / orange).
- The breadcrumb key, for example `user › 42 › update`.
- An `⚡ Optimistic` badge when an optimistic update is in progress.
- A `Retries: N` badge when the mutation has been retried.

#### Column 2: Mutation inspector

The inspector shows full detail for the selected mutation:

- **Status**: current status badge.
- **Actions**: `Retry` re-triggers the mutation on the device.
- **Variables**: input payload, expandable.
- **Error**: error object, expandable. Present only when the mutation has failed.
- **Rollback context**: snapshot captured before the optimistic update.
- **Metadata**: timestamps and retry count.

Large payloads are fetched in chunks from the device and reassembled locally.

#### Column 3: Secondary tabs

**Timeline**: chronological log of all session events. Filterable by kind. The stream can be paused or cleared.

**Widget Tree**: planned.

**Data Diff**: side-by-side comparison of mutation input and output. The **BEFORE** column shows the variables submitted to the mutator. The **AFTER** column shows the server result. The column header is green on success and red on error. Both columns render as interactive JSON trees.

### Mutation Inspector

A standalone tab that allocates the full panel width to mutation detail. Useful when inspecting large payloads or extended error traces without the mutations list occupying column space.

### Network Activity Monitor

The Network Activity Monitor records every query fetch. Each screen section serves a distinct purpose:

- **Stats strip**: total request count, average duration in milliseconds, and error rate since session start.
- **Active**: in-flight fetches with query key and elapsed time.
- **Recent**: completed fetches with key, duration, payload size, and status.

### Performance Metrics

The Performance Metrics tab accumulates fetch statistics per query key for the entire session:

- **Summary cards**: total fetches, average latency, error rate, and unique query count.
- **Per-key table**: sortable by fetch count, average duration, error count, or last active time.
- The **Clear** button in the header resets all accumulated statistics without restarting the app.

### Dependency Graph

The Dependency Graph visualizes relationships between mutations and the queries they invalidate.

Edges are inferred at runtime. When a mutation settles and a query is invalidated within 500 ms of that event, an edge is drawn between them. No annotation is required in application code.

When `QoraOptions.dependsOn` is set on a query, the graph also renders query-to-query edges. These edges are derived from `QueryEvent.dependsOnKey` and reflect the declared dependency structure. They do not rely on timing heuristics.

Graph properties:

- Mutation nodes are red; query nodes are blue.
- Edges are Bézier arrows.
- Pan and zoom are handled by `InteractiveViewer`. Pinch gestures work on touch screens; the scroll wheel works on desktop.
- Tapping a node highlights its connections and opens a detail panel with related keys.
- The filter bar searches nodes by key substring.

## Timeline

The timeline records every lifecycle event with a precise timestamp:

- `QueryCreated`: a new query was registered with the client.
- `FetchStarted`: a network request was initiated.
- `FetchError`: a request failed. Duration is included.
- `MutationStarted`: a mutation was triggered.
- `MutationSuccess`: a mutation completed. Duration is included.
- `MutationError`: a mutation failed. Duration is included.
- `OptimisticUpdate`: an optimistic update was applied before server confirmation.

The **Filter** field narrows the log by key or event type. **Pause** freezes the stream. **Clear** resets it.

## Remote commands

The extension sends commands to the running app over Dart VM Service extensions (`ext.qora.*`). Commands are dispatched to `QoraClient` via `QoraClientTrackingGateway`.

- **Refetch**: re-executes the fetcher for the given query key.
- **Invalidate**: marks the query as stale and schedules a background refetch.
- **Retry**: re-triggers a failed mutation.

---

## Large payload handling

The Dart VM Service imposes a message size limit. When a query caches a dataset that exceeds this limit, a chunked transfer protocol is used:

1. The app pushes an event containing only metadata and a `payloadId`.
2. The extension detects `hasLargePayload: true` and requests chunks sequentially via `ext.qora.getPayload`.
3. Each chunk is approximately 80 KB, base64-encoded.
4. The extension reassembles the full payload locally before rendering it.

Chunks expire after 30 seconds on the device side to bound memory usage.

## Troubleshooting

### The Qora tab does not appear in DevTools

Verify that `qora_devtools_extension` is listed under `dependencies` and not `dev_dependencies`. Verify that `QoraDevtools.setup(client)` is called before `runApp`. Restart the app after adding the dependency.

### Events are not appearing in the timeline

Confirm that the app is running in debug mode (`flutter run`). `VmTracker` does not emit events in release or profile mode.

### Commands have no effect

Commands are dispatched to the main isolate. When the app uses multiple isolates, `QoraClient` must reside in the main isolate.
