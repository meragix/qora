---

The Qora IDE extension adds a dedicated tab to Flutter DevTools. Access it through **Flutter: Open DevTools** in VS Code or **Tools > Flutter > Open DevTools** in IntelliJ.

It communicates with your running app over the Dart VM Service, which means it works on simulators, emulators, and physical devices connected via USB.

---

## Installation

Add `qora_devtools_extension` to `dependencies`, not `dev_dependencies`. The package registers VM Service extensions at startup and is removed from release builds by tree shaking.

```yaml [pubspec.yaml]
dependencies:
  qora: ^0.4.0
  qora_devtools_extension: ^0.1.0
```

Then pass a `VmTracker` to your `QoraClient`:

```dart [main.dart]
import 'package:qora_devtools_extension/qora_devtools_extension.dart';

void main() {
  final client = QoraClient(
    tracker: kDebugMode ? VmTracker() : null,
  );

  runApp(MyApp(client: client));
}
```

::callout{type="info"}
Passing `null` for `tracker` in release mode is equivalent to using `NoOpTracker`, which is the default tracker when none is provided. `VmTracker` and its dependencies are removed from release builds.
::

## Opening the extension

1. Run your app in debug mode
2. Open Flutter DevTools
   - **VS Code**: run **Flutter: Open DevTools** from the command palette
   - **IntelliJ**: **Tools → Flutter → Open DevTools**
3. Select the **Qora** tab in the DevTools tab bar

The extension connects automatically to the running isolate. No URL or port configuration needed.

## The panel

The extension is organized into six tabs.

### Queries

A live list of every query tracked by `QoraClient`, updated in real time as events arrive. For each query, the panel shows:

- The query key as a breadcrumb chip sequence (`user › 42 › profile`)
- A color-coded status badge (`success`, `loading`, `error`, `stale`, `idle`)
- Approximate payload size and last-updated timestamp
- An expandable inline JSON tree for the cached data (large payloads are loaded on demand)

Use the **Refetch** and **Invalidate** buttons on any row to trigger actions without modifying your app code. Use the filter bar at the top to search by key substring. The **Refresh** button forces a full cache snapshot sync.

### Mutations

The same three-column layout as the in-app overlay, with additional detail available in the IDE context.

#### Column 1: Mutations list

All mutations with status indicators and elapsed time. Each row shows:

- Status dot (green / blue / orange)
- Breadcrumb key, for example `user › 42 › update`
- `⚡ Optimistic` badge when an optimistic update is pending
- `Retries: N` badge when the mutation has retried

#### Column 2: Mutation inspector

Full detail for the selected mutation:

- **Status**: Current status badge
- **Actions**: `Retry` to re-trigger the mutation on the device
- **Variables**: Input payload, expandable
- **Error**: Error object if the mutation failed, expandable
- **Rollback context**: Pre-optimistic-update snapshot
- **Metadata**: Timestamps and retry count

Large payloads are loaded on demand. The extension fetches chunks from the device and reassembles them locally.

#### Column 3: Secondary tabs

**Timeline**: Chronological log of all session events. Filter by kind, pause the stream, or clear.

**Widget Tree**: Planned.

**Data Diff**: side-by-side comparison of the mutation's input and output. The **BEFORE** column shows the variables submitted to the mutator. The **AFTER** column shows the server result (green header on success, red on error). Both columns render as an interactive JSON tree.

### Mutation Inspector

A standalone tab that gives more horizontal space to the mutation detail when you want to inspect large payloads or long error traces without the list taking up column space.

### Network Activity Monitor

Tracks every query fetch in real time for diagnosing waterfalls, slow endpoints, and unexpected refetches.

The screen is split into three sections:

- **Stats strip**: Total requests, average duration (ms), and error rate since session start.
- **Active**: In-flight fetches with query key and elapsed time.
- **Recent**: Completed fetches with key, duration, payload size, and status.

### Performance Metrics

Aggregates fetch statistics per query key across the entire session.

- **Summary cards**: Total fetches, average latency, error rate, and unique query count.
- **Per-key table**: Sortable by fetch count, average duration, error count, or last active time.
- **Clear** button in the header resets all accumulated stats without restarting the app.

### Dependency Graph

Visualizes the relationship between mutations and the queries they invalidate.

The graph is inferred automatically: when a mutation settles and a query is invalidated within 500 ms, an edge is drawn between them. No annotation or configuration needed in your app code.

- Mutation nodes are shown in red, query nodes in blue.
- Edges are drawn as Bézier arrows.
- Pan and zoom with `InteractiveViewer`. Use pinch gestures on touch screens or the scroll wheel on desktop.
- Tap any node to highlight its connections and open a detail panel with related keys.
- Use the filter bar to search nodes by key substring.

## Timeline

The timeline is a flight recorder for your app's data layer. It logs every event with a precise timestamp:

- `QueryCreated`: New query registered with the client
- `FetchStarted`: Network request initiated
- `FetchError`: Request failed, with duration
- `MutationStarted`: Mutation triggered
- `MutationSuccess`: Mutation completed successfully, with duration
- `MutationError`: Mutation failed, with duration
- `OptimisticUpdate`: Optimistic update applied before server confirmation

Use the **Filter** field to narrow the log by key or event type. Use **Pause** to freeze the stream while you inspect a specific moment. Use **Clear** to reset.

## Sending commands from the IDE

The extension can send commands back to your running app without you touching a line of code.

- **Refetch**: Re-runs the fetcher for the given query key
- **Invalidate**: Marks the query as stale and triggers background refetch
- **Retry**: Re-triggers a failed mutation

These commands are sent over Dart VM Service extensions (`ext.qora.*`) registered by `qora_devtools_extension` at app startup.

## How large payloads are handled

The Dart VM Service has a message size limit. When a query caches a large dataset, the extension uses a chunked transfer protocol:

1. The app pushes a lightweight event containing only metadata and a `payloadId`
2. The IDE extension detects `hasLargePayload: true` and requests the chunks one by one via `ext.qora.getPayload`
3. Each chunk is ~80 KB, base64-encoded
4. The extension reassembles the full payload locally and displays it

Chunks expire after 30 seconds on the device side to prevent memory leaks.

## Troubleshooting

### The Qora tab does not appear in DevTools

Make sure `qora_devtools_extension` is in your `dependencies` (not `dev_dependencies`) and that you passed a `VmTracker` to your `QoraClient`. Restart the app after adding the dependency.

### Events are not showing up in the timeline

Confirm the app is running in debug mode (`flutter run`, not `flutter run --release`). The `VmTracker` does not emit events in release or profile mode.

### Commands (Refetch, Retry) have no effect

The commands are sent to the main isolate. If your app uses multiple isolates, make sure `QoraClient` lives in the main isolate.
