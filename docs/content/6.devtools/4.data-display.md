---
title: Displaying Query Data
description: How Qora renders query data in the DevTools overlay and IDE extension, and how to make your models display as structured JSON.
---

Both DevTools surfaces — the in-app overlay and the IDE extension — render cached query data as an interactive JSON tree. Whether that tree is meaningful depends on what type `T` your query returns.

---

## How data is serialized

`QoraClient` does not know your domain types. When a fetch settles, it passes the result to the tracker using the following resolution order:

| Priority | Condition                          | Result                                              |
| -------- | ---------------------------------- | --------------------------------------------------- |
| 1        | `QoraOptions.toJson` is set        | Calls your function, uses the return value          |
| 2        | `String`, `num`, `bool`, or `null` | Passed through unchanged                            |
| 3        | `List`                             | Each element is recursively serialized              |
| 4        | `Map`                              | Each value is recursively serialized                |
| 5        | Object with a `toJson()` method    | Called dynamically, result used                     |
| 6        | Everything else                    | `.toString()` — displayed as a plain string         |

The walk is recursive: a `List<Map<String, User>>` is handled correctly without any configuration, as long as `User` has a `toJson()` method.

---

## Common scenarios

### `json_serializable` and `freezed`

Both code generators produce a `toJson()` method. DevTools detects and calls it automatically.

```dart [user_repository.dart]
// @JsonSerializable() or @freezed — no extra config needed
fetchQuery<User>(
  key: ['users', userId],
  fetcher: () => api.getUser(userId),
);
```

The overlay and IDE extension will render the full object graph as an expandable tree.

---

### Raw `Map` responses

If your fetcher returns `Map<String, dynamic>` directly — for example when you decode the HTTP response manually — no configuration is needed either.

```dart [user_repository.dart]
fetchQuery<Map<String, dynamic>>(
  key: ['users', userId],
  fetcher: () async {
    final response = await http.get(Uri.parse('/users/$userId'));
    return jsonDecode(response.body) as Map<String, dynamic>;
  },
);
```

This is the zero-config approach. The downside is that the deserialization step must happen in the consuming widget or repository, not in the fetcher.

---

### Manual models without `toJson`

If your model has no `toJson()` method, DevTools will fall back to `.toString()` and display a single string node. To get structured output, pass an explicit serializer.

```dart [user_repository.dart]
fetchQuery<User>(
  key: ['users', userId],
  fetcher: () => api.getUser(userId),
  options: QoraOptions(
    toJson: (data) => (data as User).toJson(),
  ),
);
```

---

### Lists

Lists of typed objects follow the same rules. If the item type has `toJson()`, each element is serialized automatically.

```dart [posts_repository.dart]
fetchQuery<List<Post>>(
  key: ['posts'],
  fetcher: () => api.getPosts(),
  // Post has toJson() — auto-detected, no extra config
);
```

If individual items lack `toJson()`:

```dart [posts_repository.dart]
fetchQuery<List<Post>>(
  key: ['posts'],
  fetcher: () => api.getPosts(),
  options: QoraOptions(
    toJson: (data) => (data as List<Post>).map((p) => p.toJson()).toList(),
  ),
);
```

---

## The `toJson` option

`QoraOptions.toJson` accepts any function with the signature:

```dart
Object? Function(Object? data)
```

The return value must be JSON-safe: `Map<String, dynamic>`, `List`, `String`, `num`, `bool`, or `null`. Anything else will cause a runtime error when the DevTools renderer attempts to encode it.

```dart [user_repository.dart]
// Casting is safe here: T is User and data is the fetcher result
options: QoraOptions(
  toJson: (data) => (data as User).toJson(),
),
```

`toJson` is merged like all other `QoraOptions` fields: a value on `QoraClientConfig.defaultOptions` applies globally and can be overridden per query.

---

## When to use each approach

| Situation | Recommendation |
| --------- | -------------- |
| `json_serializable` or `freezed` models | Nothing to do. Auto-detected. |
| Raw `Map` / `List` from `jsonDecode` | Nothing to do. Passed through. |
| Manual model with a `toJson()` | Nothing to do. Auto-detected. |
| Model without `toJson()` | Provide `QoraOptions(toJson: ...)`. |
| Custom serialization (non-standard `toJson` signature) | Provide `QoraOptions(toJson: ...)`. |

---

## Production overhead

`_serializeForTracker` is only called when a `QoraTracker` is active. In production, `QoraClient` defaults to `NoOpTracker` — no serialization runs, zero overhead.

```dart [main.dart]
final client = QoraClient(
  tracker: kDebugMode ? tracker : null, // null → NoOpTracker
);
```
