## Objectif

**Contexte** : La connectivité mobile est intermittente par nature. Les mutations perdues pendant les périodes hors ligne créent une divergence de données et un risque opérationnel. Un système offline-first garantit la livraison éventuelle dans l'ordre de soumission sans écritures dupliquées.

Cet exemple démontre comment y parvenir en utilisant les couches de persistance et de conscience réseau de Qora, sans logique de synchronisation personnalisée.

---

## Fonctionnalités démontrées

- **`PersistQoraClient`** : cache write-through qui survit aux redémarrages de l'app via Hive
- **`hydrate()`** : chauffe le cache depuis le disque avant le rendu de la première frame ; les utilisateurs de retour ne voient jamais de spinner
- **`NetworkMode.offlineFirst`** : sert les données périmées instantanément et planifie la revalidation en arrière-plan à la reconnexion
- **`OfflineMutationQueue` avec `stopOnFirstError: true`** : rejeu FIFO qui préserve l'ordre create→update
- **`optimisticResponse` + flag `isOptimistic`** : l'UI reflète les écritures instantanément avec un indicateur "synchronisation en attente" clair
- **`FetchStatus.paused`** : expose l'état hors ligne à l'UI sans aucune gestion d'état personnalisée
- **`NetworkStatusIndicator`** : bannière hors ligne automatique sans boilerplate de widget
- **`ReconnectStrategy`** : concurrence contrôlée et jitter pour prévenir le thundering-herd à la reconnexion

---

## Points d'architecture

La décision architecturale clé est que `PersistQoraClient` est un **remplacement transparent** de `QoraClient` : il étend la classe de base et chaque API existante (`fetchQuery`, `watchQuery`, `invalidate`, ...) fonctionne inchangée. Le reste de l'app ignore la persistance.

```text
main.dart
├── HiveStorageAdapter.init()          ← ouvrir la boîte Hive
├── PersistQoraClient(storage, ...)    ← remplacement transparent de QoraClient
├── registerAllSerializers(client)     ← enregistrer les sérialiseurs Todo + List<Todo>
├── client.hydrate()                   ← injecter les données disque dans le cache avant runApp
└── QoraScope(
      client: client,
      lifecycleManager: FlutterLifecycleManager(...),    ← refetch à la reprise de l'app
      connectivityManager: FlutterConnectivityManager(), ← déclencher le rejeu à la reconnexion
      child: NetworkStatusIndicator(child: MyApp()),     ← bannière hors ligne, zéro config
    )
```

Les sérialiseurs sont regroupés dans une seule fonction `registerAllSerializers()` : cela devient la liste autoritaire de chaque type écrit sur disque, utile pour l'audit lors des migrations de modèle.

```dart [lib/core/persistence/serializers.dart]
void registerAllSerializers(PersistQoraClient client) {
  client.registerSerializer<Todo>(
    QoraSerializer(toJson: (t) => t.toJson(), fromJson: Todo.fromJson),
    name: 'Todo', // nom explicite : survit à l'obfuscation Flutter Web
  );
  client.registerSerializer<List<Todo>>(
    QoraSerializer(
      toJson: (list) => list.map((t) => t.toJson()).toList(),
      fromJson: (json) => (json as List).map(Todo.fromJson).toList(),
    ),
    name: 'List<Todo>',
  );
}
```

`OfflineMutationQueue` est configuré avec `stopOnFirstError: true` au niveau du client, car les mutations de todo ont des dépendances d'ordre (un todo doit exister avant d'être complété ou supprimé).

```dart [lib/main.dart]
final client = PersistQoraClient(
  storage: HiveStorageAdapter(boxName: 'qora_cache'),
  persistDuration: const Duration(days: 30),
  config: const QoraClientConfig(
    defaultOptions: QoraOptions(
      staleTime: Duration(minutes: 5),
      networkMode: NetworkMode.offlineFirst,
    ),
    reconnectStrategy: ReconnectStrategy(
      maxConcurrent: 3,
      jitter: Duration(milliseconds: 150),
    ),
    debugMode: kDebugMode,
  ),
);
```

## Détails d'implémentation

### Le problème d'ordre : pourquoi `stopOnFirstError` est important

Considérez cette session hors ligne :

```text
1. L'utilisateur crée Todo("Acheter du lait")   → mis en file comme PendingMutation #1
2. L'utilisateur le marque complété             → mis en file comme PendingMutation #2 (référence l'ID du todo)
3. La connectivité revient
4. Rejeu de la file : #1 se déclenche → le serveur crée le todo → retourne { id: "srv-42" }
5. Rejeu de la file : #2 se déclenche → PUT /todos/srv-42/complete → ✅
```

Si `stopOnFirstError` était `false` et que la mutation #1 échouait (ex. serveur retourne 422), la mutation #2 se déclencherait quand même — en référençant un ID inexistant sur le serveur. Avec `stopOnFirstError: true`, la mutation #2 est remise en file en tête et réessayée à la prochaine reconnexion, après résolution de la cause racine de #1.

### La boucle de retour `isOptimistic`

Quand un utilisateur crée un todo hors ligne, la mutation émet `MutationSuccess(isOptimistic: true)` : l'élément apparaît immédiatement dans la liste. Le flag `isOptimistic` pilote un visuel "synchronisation en attente" qui fournit un retour honnête et non alarmant :

```dart [lib/ui/todo_list.dart]
QoraMutationBuilder<Todo, CreateTodoInput, void>(
  mutator: (input) => api.createTodo(input),
  options: MutationOptions(
    offlineQueue: true,
    // Fournir un objet optimiste local : apparaît dans l'UI immédiatement
    optimisticResponse: (input) => Todo(
      id: 'temp-${DateTime.now().millisecondsSinceEpoch}',
      title: input.title,
      completed: false,
      isPending: true,
    ),
    onSuccess: (_, __, ___) async {
      // S'exécute pour le VRAI succès uniquement (isOptimistic: false)
      // La réponse serveur remplace l'ID temporaire : invalider pour re-trier
      context.qora.invalidate(const ['todos']);
    },
  ),
  builder: (context, mutationState, mutate) {
    final isQueued = switch (mutationState) {
      MutationSuccess(:final isOptimistic) => isOptimistic,
      _ => false,
    };

    return FloatingActionButton.extended(
      onPressed: mutationState.isPending ? null : () => mutate(input),
      icon: isQueued
          ? const Icon(Icons.schedule) // icône horloge quand mis en file hors ligne
          : const Icon(Icons.add),
      label: Text(isQueued ? 'En file : synchronisation imminente' : 'Ajouter Todo'),
    );
  },
)
```

Et dans chaque `TodoTile`, l'élément optimiste est rendu avec un style en attente discret :

```dart [lib/ui/todo_tile.dart]
ListTile(
  title: Text(
    todo.title,
    style: TextStyle(
      color: todo.isPending ? Colors.grey.shade500 : null,
      fontStyle: todo.isPending ? FontStyle.italic : null,
    ),
  ),
  leading: todo.isPending
      ? const SizedBox(
          width: 20,
          height: 20,
          child: CircularProgressIndicator(strokeWidth: 2),
        )
      : Checkbox(value: todo.completed, onChanged: (_) => _complete(todo)),
  trailing: todo.isPending
      ? Tooltip(
          message: 'Se synchronisera quand en ligne',
          child: Icon(Icons.cloud_off, size: 16, color: Colors.grey.shade400),
        )
      : null,
)
```

### Démarrage à froid : zéro spinner pour les utilisateurs de retour

`hydrate()` est attendu avant `runApp`. L'état `Success` restauré porte l'horodatage `persistedAt` original. Comme `staleTime` est de 5 minutes, un utilisateur qui ouvre l'app dans les 5 minutes de sa dernière session voit sa liste de todos **à la première frame** : sans appel réseau, sans spinner. Après 5 minutes, la liste en cache est affichée immédiatement pendant que Qora revalide en arrière-plan (SWR) :

```dart [lib/main.dart]
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();

  final storage = HiveStorageAdapter(boxName: 'qora_cache');
  await storage.init();

  final client = PersistQoraClient(storage: storage, /* ... */);
  registerAllSerializers(client);
  await client.hydrate(); // ← chauffer le cache avant la première frame

  runApp(QoraScope(client: client, child: const MyApp()));
}
```

### Gestion de l'état `FetchStatus.paused`

`QoraBuilder` expose `fetchStatus` en tant que troisième argument du builder. Quand l'appareil est hors ligne et qu'une requête n'a pas de données en cache, l'état est `Initial` ou `Loading(previousData: null)` avec `FetchStatus.paused`. Cela est distinct de `Loading(previousData: null)` + `FetchStatus.fetching` (chargement actif) et doit être rendu différemment :

```dart [lib/ui/todo_screen.dart]
QoraBuilder<List<Todo>>(
  queryKey: const ['todos'],
  fetcher: api.getTodos,
  builder: (context, state, fetchStatus) {
    // Cas spécial : hors ligne + aucun cache du tout
    if (state is Initial && fetchStatus == FetchStatus.paused) {
      return const Center(
        child: Column(
          mainAxisAlignment: MainAxisAlignment.center,
          children: [
            Icon(Icons.wifi_off, size: 48, color: Colors.grey),
            SizedBox(height: 16),
            Text('Pas de connexion : connectez-vous pour charger vos todos'),
          ],
        ),
      );
    }

    return switch (state) {
      Initial() || Loading(previousData: null) => const CircularProgressIndicator(),
      Failure(:final error, previousData: null) => ErrorScreen(error),
      // Données disponibles : Success, ou Loading/Failure avec previousData en cache
      _ => TodoList(
          todos: state.dataOrNull ?? [],
          isRefreshing: fetchStatus == FetchStatus.fetching,
          isOffline: fetchStatus == FetchStatus.paused,
        ),
    };
  },
)
```
