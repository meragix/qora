`QoraBuilder<T>` est le moyen principal de lier les données serveur à l'UI Flutter. Il gère le cycle de vie complet des requêtes : récupération, cache, réactivité, revalidation en arrière-plan, avec une seule déclaration de widget.

---

## Utilisation de base

```dart [user_list_screen.dart]
QoraBuilder<List<User>>(
  queryKey: ['users'],
  fetcher: () => api.getUsers(),
  builder: (context, state, fetchStatus) {
    return switch (state) {
      Initial() => const SizedBox.shrink(),
      Loading(:final previousData) => previousData != null
          ? Stack(children: [UserList(previousData), const LinearProgressIndicator()])
          : const CircularProgressIndicator(),
      Success(:final data) => UserList(data),
      Failure(:final error, :final previousData) => previousData != null
          ? Column(children: [UserList(previousData), ErrorBanner(error)])
          : ErrorScreen(error),
    };
  },
)
```

Le builder reçoit trois arguments :

1. **`context`** : le `BuildContext`
2. **`state`** : le `QoraState<T>` courant (ce que vous avez)
3. **`fetchStatus`** : le `FetchStatus` courant (ce que Qora fait en ce moment)

La correspondance de motifs exhaustive sur le `state` scellé est l'approche idiomatique. Utiliser `fetchStatus` pour les indicateurs secondaires (barres de progression, chips hors-ligne) sans modifier la vue d'état principale.

---

## Fonctionnement

Au montage, `QoraBuilder` :

1. **S'abonne** à `client.watchState<T>(queryKey)` : un stream d'observation uniquement qui émet chaque mise à jour d'état pour cette clé (y compris les mises à jour déclenchées par d'autres widgets ou services en arrière-plan).
2. **Récupère** via `client.fetchQuery(...)` si `enabled` est `true`. Le client gère automatiquement la déduplication, le cache et le stale-while-revalidate.

Quand la requête est **invalidée externalement** (via `client.invalidate()`, une reprise du cycle de vie ou un événement de reconnexion), l'état passe à `Loading(previousData: data)`. `QoraBuilder` détecte ce pattern dans le stream et déclenche un `fetchQuery` frais. La déduplication du client garantit qu'une seule requête réseau se déclenche même si plusieurs widgets sont abonnés à la même clé.

---

## Props

| Prop               | Type                                                         | Défaut    | Description                                                                              |
| ------------------ | ------------------------------------------------------------ | --------- | ---------------------------------------------------------------------------------------- |
| `queryKey`         | `Object` (`List` ou `QoraKey`)                               | requis    | Clé de cache. La changer déclenche un réabonnement et un fetch frais.                   |
| `fetcher`          | `Future<T> Function()`                                       | requis    | Fonction asynchrone qui retourne les données. Doit lever une exception en cas d'échec.  |
| `builder`          | `Widget Function(BuildContext, QoraState<T>, FetchStatus)`   | requis    | Construit l'arbre de widgets depuis l'état courant et le statut de fetch.               |
| `options`          | `QoraOptions?`                                               | `null`    | Config par requête fusionnée sur `QoraClientConfig.defaultOptions`.                     |
| `enabled`          | `bool`                                                       | `true`    | Quand `false`, aucun fetch n'est déclenché. Le widget observe quand même l'état en cache. |
| `keepPreviousData` | `bool`                                                       | `false`   | Enrichit les états `Loading`/`Failure` avec les dernières données `Success` de ce widget. |
| `client`           | `QoraClient?`                                                | `null`    | Surcharge le client du `QoraScope`. Utile pour les tests ou les scopes imbriqués.       |

## Patterns

### Traiter les quatre états

Utiliser `.when()` pour la correspondance exhaustive ou `.maybeWhen()` quand seuls certains états sont pertinents :

```dart [user_screen.dart]
QoraBuilder<User>(
  queryKey: ['users', userId],
  fetcher: () => api.getUser(userId),
  builder: (context, state, fetchStatus) {
    return state.when(
      onInitial: () => const SizedBox.shrink(),
      onLoading: (previousData) => previousData != null
          ? UserCard(user: previousData, isRefreshing: true)
          : const CircularProgressIndicator(),
      onSuccess: (user, updatedAt) => UserCard(user: user),
      onError: (error, stackTrace, previousData) => ErrorScreen(
        message: error.toString(),
        onRetry: () => context.qora.invalidate(['users', userId]),
      ),
    );
  },
)
```

### UI adaptée au réseau avec `FetchStatus`

`FetchStatus` est le second axe aux côtés de `QoraState`. L'utiliser pour afficher un indicateur de rafraîchissement en arrière-plan ou un message "hors-ligne : en attente" sans modifier la vue d'état principale :

```dart [user_list_screen.dart]
QoraBuilder<List<User>>(
  queryKey: ['users'],
  fetcher: api.getUsers,
  builder: (context, state, fetchStatus) {
    return switch (state) {
      Initial() when fetchStatus == FetchStatus.paused =>
          const Center(child: Text('Hors-ligne : connectez-vous pour charger')),
      Initial() || Loading() =>
          const CircularProgressIndicator(),
      Success(:final data) => Column(
          children: [
            if (fetchStatus == FetchStatus.fetching)
              const LinearProgressIndicator(),
            if (fetchStatus == FetchStatus.paused)
              const OfflineSyncChip(),
            UserList(data),
          ],
        ),
      Failure(:final error) => ErrorScreen(error),
    };
  },
)
```

| Valeur `FetchStatus`   | Signification                                                    |
| ---------------------- | ---------------------------------------------------------------- |
| `FetchStatus.fetching` | Une requête réseau est en cours.                                 |
| `FetchStatus.paused`   | Hors-ligne : le fetch est en file et sera rejoué à la reconnexion. |
| `FetchStatus.idle`     | Aucun fetch actif (les données sont fraîches ou l'état est terminal). |

:::tip
`FetchStatus.paused` n'apparaît que quand `QoraOptions.networkMode` est `NetworkMode.online` (le défaut) et que l'appareil n'a pas de connectivité. Avec `NetworkMode.always`, les fetchs s'exécutent quelle que soit la connectivité et cet état n'est jamais émis.
:::

### Requêtes conditionnelles (`enabled`)

Empêcher le fetch jusqu'à ce qu'une dépendance soit disponible :

```dart [profile_screen.dart]
QoraBuilder<Profile>(
  queryKey: ['profiles', userId],
  fetcher: () => api.getProfile(userId!),
  enabled: userId != null,   // Pas de fetch tant que userId n'est pas résolu
  builder: (context, state, fetchStatus) { ... },
)
```

Quand `enabled` passe de `false` à `true` (via `setState`), un fetch est déclenché automatiquement.

---

### Pagination sans scintillement (`keepPreviousData`)

Quand la clé de requête change (ex. numéro de page incrémenté), la nouvelle clé démarre à l'état `Initial`, provoquant un spinner plein écran perturbant. `keepPreviousData` évite cela en portant les dernières données réussies dans la phase de chargement de la nouvelle clé :

```dart [paginated_posts_screen.dart]
QoraBuilder<List<Post>>(
  queryKey: ['posts', currentPage],
  fetcher: () => api.getPosts(page: currentPage),
  keepPreviousData: true,
  builder: (context, state, fetchStatus) {
    // dataOrNull retourne les données de Success, Loading.previousData, ou Failure.previousData
    final posts = state.dataOrNull ?? [];
    return Stack(
      children: [
        PostList(posts: posts),
        if (state.isLoading) const LinearProgressIndicator(),
      ],
    );
  },
)
```

:::tip
`keepPreviousData` comble uniquement l'écart : si le client fournit déjà `previousData` dans l'état, cette valeur est prioritaire. Le cache local du widget est purement un fallback.
:::

---

### Afficher un horodatage de fraîcheur

`Success` porte `updatedAt` : le moment où les données ont été récupérées pour la dernière fois :

```dart [user_screen.dart]
onSuccess: (user, updatedAt) => Column(
  children: [
    UserCard(user: user),
    Text(
      'Mis à jour ${_timeAgo(updatedAt)}',
      style: Theme.of(context).textTheme.labelSmall,
    ),
  ],
),
```

---

### Polling

Passer `refetchInterval` via `options` pour effectuer un polling à intervalle fixe :

```dart [notifications_screen.dart]
QoraBuilder<List<Notification>>(
  queryKey: ['notifications'],
  fetcher: api.getNotifications,
  options: const QoraOptions(refetchInterval: Duration(seconds: 30)),
  builder: (context, state, fetchStatus) { ... },
)
```

Le polling est actif uniquement pendant que `QoraBuilder` est dans l'arbre de widgets. Il s'arrête automatiquement au dispose.

---

## Comportement lors du changement de clé

Quand `queryKey` change (`didUpdateWidget`), `QoraBuilder` :

1. Annule l'abonnement au stream précédent.
2. S'abonne au stream de la nouvelle clé (en partant de son état courant en cache : peut être `Initial`, `Success`, etc.).
3. Déclenche un `fetchQuery` frais pour la nouvelle clé (si `enabled`).

C'est ainsi que la pagination, le changement d'onglet et les changements de filtre sont gérés automatiquement sans gestion d'état supplémentaire.

## `QoraStateBuilder` : observer sans fetcher

`QoraStateBuilder<T>` s'abonne à l'état d'une requête sans déclencher de fetch. L'utiliser quand un autre widget ou service possède le cycle de vie du fetch et que seul l'état courant doit être affiché.

```dart [notification_badge.dart]
// NotificationsScreen (un QoraBuilder) possède le fetch.
// Ce badge reflète simplement l'état en cache de la même clé.
QoraStateBuilder<List<Notification>>(
  queryKey: ['notifications'],
  builder: (context, state) {
    final count = state.dataOrNull?.length ?? 0;
    return Badge(
      label: Text('$count'),
      isLabelVisible: count > 0,
      child: const Icon(Icons.notifications),
    );
  },
)
```

### Props de `QoraStateBuilder`

| Prop       | Type                                          | Description                                      |
| ---------- | --------------------------------------------- | ------------------------------------------------ |
| `queryKey` | `Object` (`List` ou `QoraKey`)                | Clé à observer.                                  |
| `builder`  | `Widget Function(BuildContext, QoraState<T>)` | Construit l'arbre de widgets depuis l'état courant. |
| `client`   | `QoraClient?`                                 | Surcharge le client du `QoraScope`.              |

:::tip
`QoraStateBuilder` est adapté aux badges, compteurs ou affichages secondaires qui dérivent leurs données d'une requête chargée ailleurs : par exemple, un compteur d'articles dans le panier ou un badge de messages non lus dans une barre d'application.
:::
