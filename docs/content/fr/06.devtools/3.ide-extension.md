L'extension IDE Qora ajoute un onglet dédié à Flutter DevTools. Elle est accessible via **Flutter: Open DevTools** dans VS Code ou **Tools > Flutter > Open DevTools** dans IntelliJ.

L'extension communique avec l'app en cours d'exécution via le Dart VM Service. Elle est compatible avec les simulateurs, émulateurs et appareils physiques connectés via USB.

<!-- ## Installation

Ajouter `qora_devtools_extension` dans `dependencies`, pas dans `dev_dependencies`. Le package enregistre les extensions VM Service au démarrage. Le tree shaking supprime automatiquement le code concerné des builds de release.

```yaml [pubspec.yaml]
dependencies:
  qora: ^1.0.0
  qora_devtools_extension: ^1.0.0
```

Appeler `QoraDevtools.setup` une fois dans `main`, avant `runApp` :

```dart [main.dart]
import 'package:flutter/foundation.dart';
import 'package:qora_devtools_extension/qora_devtools_extension.dart';

void main() {
  final client = QoraClient();

  if (kDebugMode) QoraDevtools.setup(client);

  runApp(MyApp(client: client));
}
```

`QoraDevtools.setup` effectue trois opérations en séquence : il initialise un `LazyPayloadManager`, construit un `VmTracker` soutenu par ce manager, et enregistre tous les gestionnaires d'extensions VM Service `ext.qora.*`.

### Avec l'overlay intégré

Quand l'[overlay intégré](/devtools/overlay) est également utilisé, passer l'`OverlayTracker` via `additionalTrackers`. Les deux surfaces reçoivent les mêmes événements depuis un `MultiTracker` partagé :

```dart [main.dart]
import 'package:flutter/foundation.dart';
import 'package:qora_devtools_extension/qora_devtools_extension.dart';
import 'package:qora_devtools_overlay/qora_devtools_overlay.dart';

void main() {
  final overlay = OverlayTracker();
  final client  = QoraClient();

  if (kDebugMode) {
    QoraDevtools.setup(client, additionalTrackers: [overlay]);
  }

  runApp(
    QoraInspector(
      tracker: overlay,
      client: kDebugMode ? client : null,
      child: MyApp(client: client),
    ),
  );
}
```

::callout{type="info"}
Chaque tracker passé via `additionalTrackers` est enveloppé dans un `MultiTracker`. Les trackers reçoivent les événements dans l'ordre de déclaration.
::

### Câblage manuel

Quand un contrôle direct sur les composants individuels est requis (par exemple, pour partager un seul `LazyPayloadManager` entre plusieurs clients ou définir une taille de ring-buffer personnalisée), les composants peuvent être câblés individuellement :

```dart [main.dart]
import 'package:qora_devtools_extension/qora_devtools_extension.dart';

void main() {
  final lazy    = LazyPayloadManager();
  final tracker = VmTracker(lazyPayloadManager: lazy, maxBuffer: 1000);
  final client  = QoraClient();

  client.setTracker(tracker);

  ExtensionRegistrar(
    handlers: ExtensionHandlers(
      gateway: QoraClientTrackingGateway(client),
      lazyPayloadManager: lazy,
    ),
  ).registerAll();

  runApp(MyApp(client: client));
}
```

> L'instance de `LazyPayloadManager` passée à `VmTracker` doit être identique à celle passée à `ExtensionHandlers`. Les deux composants la partagent pour coordonner le stockage et la récupération de payloads en chunks.

## Ouverture de l'extension

1. Lancer l'app en mode debug.
2. Ouvrir Flutter DevTools.
   - **VS Code** : exécuter **Flutter: Open DevTools** depuis la palette de commandes.
   - **IntelliJ** : **Tools > Flutter > Open DevTools**.
3. Sélectionner l'onglet **Qora** dans la barre d'onglets DevTools.

L'extension se connecte à l'isolate en cours d'exécution sans nécessiter de configuration d'URL ou de port.

## Le panneau

Le panneau de l'extension est organisé en six onglets.

### Queries

L'onglet Queries affiche toutes les requêtes suivies par `QoraClient`, mis à jour à l'arrivée des événements. Chaque ligne montre :

- La clé de requête rendue sous forme de séquence de breadcrumb (`user › 42 › profile`).
- Un badge de statut (`success`, `loading`, `error`, `stale`, `idle`).
- La taille approximative du payload et l'horodatage de dernière mise à jour.
- Un arbre JSON inline extensible. Les grands payloads sont récupérés à la demande via le protocole de transfert en chunks.

Les boutons **Refetch** et **Invalidate** sur chaque ligne déclenchent l'action correspondante sur l'appareil. La barre de filtre recherche par sous-chaîne de clé. Le bouton **Refresh** force une synchronisation complète du snapshot du cache.

### Mutations

L'onglet Mutations utilise une disposition trois colonnes.

#### Colonne 1 : Liste des mutations

Chaque ligne montre :

- Un indicateur de statut (vert / bleu / orange).
- La clé breadcrumb, par exemple `user › 42 › update`.
- Un badge `⚡ Optimistic` quand une mise à jour optimiste est en cours.
- Un badge `Retries: N` quand la mutation a été relancée.

#### Colonne 2 : Inspecteur de mutation

L'inspecteur affiche les détails complets de la mutation sélectionnée :

- **Statut** : badge de statut actuel.
- **Actions** : `Retry` re-déclenche la mutation sur l'appareil.
- **Variables** : payload d'entrée, extensible.
- **Erreur** : objet d'erreur, extensible. Présent uniquement quand la mutation a échoué.
- **Contexte de rollback** : snapshot capturé avant la mise à jour optimiste.
- **Métadonnées** : horodatages et nombre de retries.

Les grands payloads sont récupérés en chunks depuis l'appareil et réassemblés localement.

#### Colonne 3 : Onglets secondaires

**Timeline** : journal chronologique de tous les événements de session. Filtrable par type. Le stream peut être mis en pause ou effacé.

**Arbre de widgets** : prévu.

**Data Diff** : comparaison côte à côte de l'entrée et de la sortie de la mutation. La colonne **BEFORE** montre les variables soumises au mutator. La colonne **AFTER** montre le résultat du serveur. L'en-tête de colonne est vert en cas de succès et rouge en cas d'erreur. Les deux colonnes se rendent sous forme d'arbres JSON interactifs.

### Inspecteur de mutation

Un onglet autonome qui alloue toute la largeur du panneau au détail de mutation. Utile pour inspecter de grands payloads ou des traces d'erreur étendues sans que la liste des mutations occupe l'espace des colonnes.

### Moniteur d'activité réseau

Le Moniteur d'activité réseau enregistre chaque fetch de requête. Chaque section de l'écran sert un objectif distinct :

- **Bande de stats** : nombre total de requêtes, durée moyenne en millisecondes, et taux d'erreur depuis le début de la session.
- **Actif** : fetches en cours avec clé de requête et temps écoulé.
- **Récent** : fetches complétés avec clé, durée, taille du payload et statut.

### Métriques de performance

L'onglet Métriques de performance accumule les statistiques de fetch par clé de requête pour toute la session :

- **Cartes de synthèse** : total des fetches, latence moyenne, taux d'erreur et nombre de requêtes uniques.
- **Tableau par clé** : triable par nombre de fetches, durée moyenne, nombre d'erreurs ou dernière heure active.
- Le bouton **Clear** dans l'en-tête réinitialise toutes les statistiques accumulées sans redémarrer l'app.

### Graphe de dépendances

Le Graphe de dépendances visualise les relations entre les mutations et les requêtes qu'elles invalident.

Les arêtes sont inférées à l'exécution. Quand une mutation se règle et qu'une requête est invalidée dans les 500 ms de cet événement, une arête est tracée entre eux. Aucune annotation n'est requise dans le code applicatif.

Quand `QoraOptions.dependsOn` est défini sur une requête, le graphe rend également des arêtes requête-vers-requête. Ces arêtes sont dérivées de `QueryEvent.dependsOnKey` et reflètent la structure de dépendance déclarée. Elles ne s'appuient pas sur les heuristiques de timing.

Propriétés du graphe :

- Les nœuds de mutation sont rouges ; les nœuds de requête sont bleus.
- Les arêtes sont des flèches de Bézier.
- Le pan et le zoom sont gérés par `InteractiveViewer`. Les gestes de pincement fonctionnent sur les écrans tactiles ; la molette de défilement fonctionne sur desktop.
- Appuyer sur un nœud met en évidence ses connexions et ouvre un panneau de détail avec les clés associées.
- La barre de filtre recherche les nœuds par sous-chaîne de clé.

## Timeline

La timeline enregistre chaque événement de cycle de vie avec un horodatage précis :

- `queryCreated` : une nouvelle clé de requête a été insérée dans le cache.
- `fetchStarted` : une requête réseau a été initiée.
- `fetchSuccess` : un fetch s'est complété avec succès. La durée est incluse.
- `fetchError` : un fetch a échoué. La durée est incluse.
- `queryInvalidated` : une requête a été marquée périmée et sera re-fetchée au prochain accès.
- `queryRemoved` : une clé de requête a été évincée du cache (GC ou suppression explicite).
- `queryMarkedStale` : une requête a été marquée périmée sans déclencher de refetch en arrière-plan.
- `queryCancelled` : un fetch a été annulé avant ou pendant la requête.
- `mutationStarted` : une mutation a été déclenchée.
- `mutationSuccess` : une mutation s'est complétée. La durée est incluse.
- `mutationError` : une mutation a échoué. La durée est incluse.
- `optimisticUpdate` : une écriture de cache optimiste a été appliquée avant la confirmation du serveur.
- `cacheCleared` : l'intégralité du cache a été vidée.

Le champ **Filter** restreint le journal par clé ou type d'événement. **Pause** fige le stream. **Clear** le réinitialise.

## Commandes à distance

L'extension envoie des commandes à l'app en cours d'exécution via les extensions VM Service Dart (`ext.qora.*`). Les commandes sont dispatchées à `QoraClient` via `QoraClientTrackingGateway`.

- **Refetch** : ré-exécute le fetcher pour la clé de requête donnée.
- **Invalidate** : marque la requête comme périmée et planifie un refetch en arrière-plan.
- **Retry** : re-déclenche une mutation échouée.

---

## Gestion des grands payloads

Le Dart VM Service impose une limite de taille de message. Quand une requête met en cache un dataset dépassant cette limite, un protocole de transfert en chunks est utilisé :

1. L'app pousse un événement contenant uniquement des métadonnées et un `payloadId`.
2. L'extension détecte `hasLargePayload: true` et demande les chunks séquentiellement via `ext.qora.getPayload`.
3. Chaque chunk fait approximativement 80 Ko, encodé en base64.
4. L'extension réassemble le payload complet localement avant de le rendre.

Les chunks expirent après 30 secondes côté appareil pour limiter l'usage mémoire.

## Dépannage

### L'onglet Qora n'apparaît pas dans DevTools

Vérifier que `qora_devtools_extension` est listé sous `dependencies` et non `dev_dependencies`. Vérifier que `QoraDevtools.setup(client)` est appelé avant `runApp`. Redémarrer l'app après l'ajout de la dépendance.

### Les événements n'apparaissent pas dans la timeline

Confirmer que l'app s'exécute en mode debug (`flutter run`). `VmTracker` n'émet pas d'événements en mode release ou profile.

### Les commandes n'ont aucun effet

Les commandes sont dispatchées à l'isolate principal. Quand l'app utilise plusieurs isolates, `QoraClient` doit résider dans l'isolate principal. -->
