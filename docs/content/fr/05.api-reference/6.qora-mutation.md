## `MutationController`

Le contrôleur de cycle de vie central pour une seule mutation.

| Paramètre de type | Rôle                                                           |
| ----------------- | -------------------------------------------------------------- |
| `TData`           | La valeur retournée par le mutator en cas de succès            |
| `TVariables`      | La valeur passée à `mutate()`                                  |
| `TContext`        | Le snapshot retourné par `onMutate` (utilisé pour le rollback) |

### Constructeur

```dart
MutationController({
  required MutatorFunction<TData, TVariables> mutator,
  MutationOptions<TData, TVariables, TContext>? options,
  MutationTracker? tracker,
  Map<String, Object?>? metadata,
})
```

```dart [post_repository.dart]
final controller = MutationController<Post, String, List<Post>?>(
  mutator: (title) => api.createPost(title),
  options: MutationOptions(
    onMutate: (title) async => client.getQueryData(['posts']),
    onError: (err, _, prev) async => client.restoreQueryData(['posts'], prev),
    onSuccess: (_, __, ___) async => client.invalidate(['posts']),
  ),
  tracker: client,
  metadata: {'category': 'content', 'resource': 'post'},
);
```

### Propriétés

#### `id`

```dart
final String id
```

Identifiant unique pour cette instance de contrôleur. Format : `mutation_N` (incrémental monotone). Utilisé comme clé dans `QoraClient.activeMutations` et `MutationUpdate.mutatorId`.

#### `mutator`

```dart
final MutatorFunction<TData, TVariables> mutator
```

La fonction async qui effectue l'écriture. Nommée `mutator` pour faire écho à `fetcher` dans `fetchQuery`.

#### `options`

```dart
final MutationOptions<TData, TVariables, TContext>? options
```

Callbacks de cycle de vie et configuration de retry. Voir `MutationOptions` ci-dessous.

#### `tracker`

```dart
final MutationTracker? tracker
```

Tracker optionnel (typiquement `QoraClient`) qui reçoit les notifications de changement d'état pour l'observabilité DevTools. Quand `null`, le contrôleur opère en mode standalone.

#### `metadata`

```dart
final Map<String, Object?>? metadata
```

Paires clé-valeur arbitraires transmises telles quelles à chaque `MutationUpdate` émis par ce contrôleur. Utiliser pour l'étiquetage DevTools sans modifier le schéma central.

#### `state`

```dart
MutationState<TData, TVariables> get state
```

Le snapshot d'état synchrone actuel.

#### `stream`

```dart
Stream<MutationState<TData, TVariables>> get stream
```

Un stream par abonné qui émet **immédiatement** l'état actuel à l'écoute (capture synchrone), puis transmet toutes les transitions suivantes.

Supporte plusieurs abonnés concurrents. Chaque abonnement obtient son propre stream soutenu par le contrôleur broadcast partagé : aucun événement n'est perdu entre `listen()` et le premier microtask.

### Méthodes

#### `mutate`

```dart
Future<TData?> mutate(TVariables variables)
```

Exécute la mutation. Séquence d'états :

1. Appelle `onMutate(variables)` → le résultat devient `TContext`
2. → `MutationPending`
3. Appelle `mutator(variables)` (avec retry optionnel)
4. → `MutationSuccess` + `onSuccess` + `onSettled`, ou → `MutationFailure` + `onError` + `onSettled`

Retourne `TData` en cas de succès, `null` en cas d'échec. Les erreurs ne sont **jamais** re-levées à l'appelant : elles sont capturées dans l'état `MutationFailure`.

**Lève** `StateError` si le contrôleur a été disposé.

#### `reset`

```dart
void reset()
```

Transition vers `MutationIdle`. Supprime également ce contrôleur de `tracker.activeMutations`.

**Lève** `StateError` si le contrôleur a été disposé.

#### `dispose`

```dart
void dispose()
```

Libère toutes les ressources et ferme le stream interne. Supprime silencieusement le contrôleur du tracker (aucun événement émis). Doit être appelé quand le contrôleur n'est plus nécessaire.

Après `dispose`, tout appel à `mutate` ou `reset` lève `StateError`.

## `MutationOptions`

Callbacks de cycle de vie et configuration de retry par mutation.

### Constructeur

```dart
const MutationOptions({
  Future<TContext?> Function(TVariables variables)? onMutate,
  Future<void> Function(TData data, TVariables variables, TContext? context)? onSuccess,
  Future<void> Function(Object error, TVariables variables, TContext? context)? onError,
  Future<void> Function(TData? data, Object? error, TVariables variables, TContext? context)? onSettled,
  int retryCount = 0,
  Duration retryDelay = const Duration(seconds: 1),
})
```

### Propriétés

#### `onMutate`

```dart
Future<TContext?> Function(TVariables variables)? onMutate
```

Appelé avant le mutator. Retourne `TContext` : le snapshot optimiste passé à `onError`/`onSettled` pour le rollback. Si ce callback lève, le mutator **n'est pas appelé** et le contrôleur transite directement vers `MutationFailure`.

#### `onSuccess`

```dart
Future<void> Function(TData data, TVariables variables, TContext? context)? onSuccess
```

Appelé après un mutator réussi, avant `onSettled`.

#### `onError`

```dart
Future<void> Function(Object error, TVariables variables, TContext? context)? onError
```

Appelé après l'échec du mutator (après tous les retries), avant `onSettled`. Utiliser `context` (le snapshot de `onMutate`) pour effectuer un rollback via `client.restoreQueryData`.

#### `onSettled`

```dart
Future<void> Function(TData? data, Object? error, TVariables variables, TContext? context)? onSettled
```

Appelé après chaque résultat : succès ou échec. `data` est non-null en cas de succès ; `error` est non-null en cas d'échec.

#### `retryCount`

```dart
final int retryCount
```

Nombre de tentatives de retry après l'échec initial. Défaut : `0` (pas de retry). Chaque retry utilise un backoff exponentiel : `retryDelay * 2^tentative`.

#### `retryDelay`

```dart
final Duration retryDelay
```

Délai de base pour le premier retry. Défaut : `Duration(seconds: 1)`. Les retries suivants doublent : `1s, 2s, 4s, ...`

## `MutationState`

Hiérarchie de classes sealed représentant tous les états possibles d'une mutation.

```text
sealed MutationState
  ├── MutationIdle
  ├── MutationPending   { variables }
  ├── MutationSuccess   { data, variables }
  └── MutationFailure   { error, stackTrace?, variables }
```

### Getters communs

Disponibles sur toutes les variantes :

| Getter            | Type          | Description                                    |
| ----------------- | ------------- | ---------------------------------------------- |
| `isIdle`          | `bool`        | `true` dans `MutationIdle`                     |
| `isPending`       | `bool`        | `true` dans `MutationPending`                  |
| `isSuccess`       | `bool`        | `true` dans `MutationSuccess`                  |
| `isError`         | `bool`        | `true` dans `MutationFailure`                  |
| `dataOrNull`      | `TData?`      | Non-null uniquement dans `MutationSuccess`      |
| `errorOrNull`     | `Object?`     | Non-null uniquement dans `MutationFailure`      |
| `variablesOrNull` | `TVariables?` | `null` uniquement dans `MutationIdle`           |

### `when`

```dart
void when({
  void Function()? onIdle,
  void Function(TVariables variables)? onPending,
  void Function(TData data, TVariables variables)? onSuccess,
  void Function(Object error, StackTrace? stackTrace, TVariables variables)? onError,
})
```

Appelle un callback pour la variante correspondante. Les variantes non gérées sont ignorées silencieusement.

### `maybeWhen`

```dart
R maybeWhen<R>({
  required R Function() orElse,
  R Function()? onIdle,
  R Function(TVariables)? onPending,
  R Function(TData, TVariables)? onSuccess,
  R Function(Object, StackTrace?, TVariables)? onError,
})
```

Retourne la valeur du callback correspondant, ou `orElse()` si aucun ne correspond.

### `fold` (extension)

```dart
R fold<R>({
  required R Function() onIdle,
  required R Function(TVariables) onPending,
  required R Function(TData, TVariables) onSuccess,
  required R Function(Object, StackTrace?, TVariables) onError,
})
```

Mapper exhaustif : le compilateur impose que chaque variante soit gérée.

```dart [create_post_screen.dart]
final label = state.fold(
  onIdle: () => 'Créer',
  onPending: (_) => 'Création...',
  onSuccess: (post, _) => 'Créé : ${post.title}',
  onError: (err, _, __) => 'Erreur : $err',
);
```

### `status` (extension)

```dart
MutationStatus get status
```

Retourne la valeur d'enum `MutationStatus` à grain grossier : `idle | pending | success | error`.

## `MutationStatus`

```dart
enum MutationStatus { idle, pending, success, error }
```

Statut à grain grossier utilisé par `MutationUpdate` pour éviter les problèmes d'effacement de type générique au niveau du bus d'événements.

## `MutationUpdate`

Snapshot sans type d'une transition d'état de mutation. Émis par `QoraClient.mutationEvents` à chaque changement d'état de `MutationController`.

### Propriétés

| Propriété   | Type                       | Description                                                       |
| ----------- | -------------------------- | ----------------------------------------------------------------- |
| `mutatorId` | `String`                   | Identifiant unique du contrôleur (`mutation_N`)                   |
| `status`    | `MutationStatus`           | Statut à grain grossier au moment de l'émission                   |
| `data`      | `Object?`                  | Données de succès (type effacé). Non-null en cas de succès.       |
| `error`     | `Object?`                  | Erreur. Non-null en cas d'échec.                                  |
| `variables` | `Object?`                  | Variables passées à `mutate()`. `null` quand idle.                |
| `timestamp` | `DateTime`                 | Quand cet événement a été émis                                    |
| `metadata`  | `Map<String, Object?>?`    | Transmis depuis `MutationController.metadata`                     |

### Getters

| Getter       | Type   | Description                                                                             |
| ------------ | ------ | --------------------------------------------------------------------------------------- |
| `isIdle`     | `bool` | `status == MutationStatus.idle`                                                         |
| `isPending`  | `bool` | `status == MutationStatus.pending`                                                      |
| `isSuccess`  | `bool` | `status == MutationStatus.success`                                                      |
| `isError`    | `bool` | `status == MutationStatus.error`                                                        |
| `isFinished` | `bool` | `isSuccess \|\| isError` : utilisé par `QoraClient` pour purger automatiquement `activeMutations` |

## `MutationTracker`

Interface abstraite implémentée par `QoraClient`. Découple `MutationController` du client pour éviter les imports circulaires.

```dart
abstract interface class MutationTracker {
  void trackMutation<TData, TVariables>(
    String id,
    MutationState<TData, TVariables> state, {
    Map<String, Object?>? metadata,
  });

  void untrackMutation(String id);
}
```

Implémenter `MutationTracker` pour construire des DevTools personnalisés, un middleware de journalisation ou des espions de test :

```dart [mutation_logger.dart]
class MutationLogger implements MutationTracker {
  @override
  void trackMutation<TData, TVariables>(
    String id,
    MutationState<TData, TVariables> state, {
    Map<String, Object?>? metadata,
  }) {
    print('[$id] → ${state.runtimeType} ${metadata ?? ''}');
  }

  @override
  void untrackMutation(String id) => print('[$id] disposé');
}
```

## `QoraClient` : Observabilité des mutations

`QoraClient` implémente `MutationTracker` et expose deux surfaces pour observer les mutations suivies.

### `mutationEvents`

```dart
Stream<MutationUpdate> get mutationEvents
```

Stream broadcast en temps réel de chaque `MutationUpdate` de tous les contrôleurs suivis. Émet à chaque transition d'état : y compris `MutationIdle` (reset) et `MutationFailure`.

```dart [app_error_handler.dart]
client.mutationEvents.listen((event) {
  if (event.isError) {
    logger.error(
      '[${event.mutatorId}] Mutation échouée',
      event.error,
      event.metadata,
    );
  }
});
```

:::tip
`mutationEvents` est un stream broadcast : les événements émis avant votre abonnement sont perdus. Pour attraper les mutations démarrées avant l'abonnement, lire d'abord `activeMutations`.
:::

### `activeMutations`

```dart
Map<String, MutationUpdate> get activeMutations
```

Snapshot non modifiable de toutes les mutations **actuellement en cours** (pending), indexées par `MutationController.id`.

- Entrée **ajoutée** quand un contrôleur transite vers `MutationPending`
- Entrée **purgée automatiquement** quand la mutation se termine (`MutationSuccess` ou `MutationFailure`)
- Entrée **supprimée** quand le contrôleur revient à `MutationIdle` ou est disposé

```dart [devtools_panel.dart]
// Pattern de connexion tardive DevTools :
// 1. Lire le snapshot pour voir ce qui est actuellement en vol
final pending = client.activeMutations;

// 2. S'abonner au stream pour les mises à jour en temps réel (y compris les complétions)
client.mutationEvents.listen((event) {
  devtools.push(event);
});
```

`activeMutations` n'accumule jamais d'entrées "fantômes" pour les mutations terminées : c'est toujours un indicateur fiable de "en cours d'exécution".

### `debugInfo`

`QoraClient.debugInfo()` inclut l'état des mutations dans sa sortie :

```dart [main.dart]
print(client.debugInfo());
// {
//   total_queries: 5,
//   active_queries: 2,
//   inactive_queries: 3,
//   pending_requests: 0,
//   max_size: 100,
//   active_mutations: 1,   // ← mutations actuellement en attente
//   keys: [...]
// }
```

## `MutatorFunction`

```dart
typedef MutatorFunction<TData, TVariables> =
    Future<TData> Function(TVariables variables);
```

Alias de type pour la fonction de mutation. Fait écho à `QueryFunction<T>` pour la cohérence avec la convention de nommage `fetcher`/`mutator` dans toute l'API.
