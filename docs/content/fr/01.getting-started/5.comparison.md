Qora, `flutter_query` et `cached_query` sont des bibliothèques de state serveur construites pour des données résidant sur un serveur distant. Provider et BLoC ciblent le state client ; les comparer à Qora confond deux domaines de problèmes distincts.

Cette page documente les différences significatives entre les trois solutions de state serveur. Pour la matrice complète des fonctionnalités, voir [Qu'est-ce que Qora ? — Comparaison des fonctionnalités](./what-is-qora.md#feature-comparison).

---

## Critères de sélection

| Si vous avez besoin de...                                       | Meilleur choix        |
| --------------------------------------------------------------- | --------------------- |
| Un cache SWR solide avec configuration minimale                 | L'un ou l'autre       |
| Mutations hors-ligne mises en file et rejouées à la reconnexion | **Qora**              |
| NetworkMode par requête (online / always / offlineFirst)        | **Qora**              |
| Contrôle du thundering-herd à la reconnexion                    | **Qora**              |
| API Hooks (useQuery, useMutation...)                            | **Qora** (qora_hooks) |
| DevTools IDE et in-app de première classe                       | **Qora**              |
| Noyau Dart pur (pas de dépendance Flutter SDK)                  | **Qora**              |
| Adaptateurs de stockage communautaires existants (Hive, SharedPrefs) | **cached_query** |
| Empreinte de dépendances minimale                               | **flutter_query**     |

## Différenciateurs de Qora

### Architecture hors-ligne

Qora est la seule bibliothèque de l'écosystème Dart/Flutter avec une architecture hors-ligne complète :

- **`NetworkMode` par requête** : déclarer si une requête s'exécute en mode `online` uniquement, `always` (base de données locale), ou `offlineFirst` (servir le cache, rafraîchir en arrière-plan quand en ligne).
- **File de mutations hors-ligne** : les écritures soumises hors-ligne sont mises en file dans l'ordre et rejouées automatiquement à la reconnexion. `stopOnFirstError` gère les opérations séquentiellement dépendantes.
- **`FetchStatus`** : un enum sur un second axe (`fetching / paused / idle`) émis indépendamment de `QoraState`, permettant à l'UI de distinguer "pas de réseau" de "chargement".
- **`ReconnectStrategy`** : contrôle le thundering-herd à la reconnexion avec un nombre maximum configurable de refetchs concurrents et un jitter aléatoire entre les lots.

```dart [user_screen.dart]
QoraBuilder<User>(
  queryKey: ['users', userId],
  fetcher: () => api.getUser(userId),
  options: const QoraOptions(networkMode: NetworkMode.offlineFirst),
  builder: (context, state, fetchStatus) {
    if (fetchStatus == FetchStatus.paused) {
      return OfflineBanner(staleData: state.dataOrNull);
    }
    return switch (state) {
      Success(:final data) => UserCard(data),
      Loading() => const CircularProgressIndicator(),
      Failure(:final error) => ErrorWidget('$error'),
      Initial() => const SizedBox.shrink(),
    };
  },
)
```

### DevTools

Qora inclut des outils de développement dédiés sans configuration :

- Inspecteur de cache en direct : clé, état, taille des données, péremption
- Timeline des mutations avec diff avant/après
- Onglet d'activité réseau avec durées de fetch et taux d'erreur
- Onglet de performance avec statistiques par requête
- Graphe de dépendances

Aucune autre bibliothèque de state serveur Dart ne fournit des outils équivalents.

### Noyau Dart pur

Le package `qora` n'a aucune dépendance Flutter. Il fonctionne dans les applications CLI, le code Dart côté serveur et les suites de tests sans importer le Flutter SDK. `qora_flutter` est une couche de liaison de widgets opt-in mince au-dessus.

### Invalidation avancée

`invalidateQueries({required QueryFilter filter})` invalide par état, options de dernière utilisation ou toute combinaison, pas seulement par préfixe de clé :

```dart [main.dart]
// Invalider toutes les requêtes en mode online qui n'ont pas encore réussi
client.invalidateQueries(
  filter: (key, state, opts) =>
      opts?.networkMode == NetworkMode.online && state is! Success,
);
```

## `cached_query` : écosystème d'adaptateurs de stockage

`cached_query` dispose d'un écosystème d'adaptateurs de stockage plus mature, avec des adaptateurs maintenus par la communauté pour Hive et SharedPreferences. `PersistQoraClient` de Qora adresse le même cas d'usage via une interface `StorageAdapter`, mais les adaptateurs pour des backends spécifiques doivent être trouvés ou implémentés séparément.

## Complémentarité avec BLoC et Riverpod

Qora, BLoC et Riverpod adressent des préoccupations différentes et sont conçus pour être utilisés ensemble :

| Couche                                                               | Outil approprié              |
| -------------------------------------------------------------------- | ---------------------------- |
| Données résidant sur un serveur (requêtes, mutations, pagination)    | **Qora**                     |
| Machines d'état UI complexes (tunnels de paiement, formulaires multi-étapes) | **BLoC**             |
| State client et DI (thème, session d'authentification, graphe de services) | **Riverpod** ou **Provider** |

Une application en production utilise couramment les trois en combinaison. Consulter les guides d'intégration pour ajouter Qora à une base de code BLoC ou Riverpod existante.

:::tip
Qora gère le state serveur. BLoC et Riverpod gèrent le state client. Les deux domaines sont complémentaires ; chaque bibliothèque opère dans son périmètre prévu.
:::
