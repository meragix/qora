Qora partage le même modèle conceptuel que TanStack Query : clés de requête, stale-while-revalidate, refetch en arrière-plan, hooks de cycle de vie des mutations et pause-et-replay hors-ligne. La plupart des concepts se traduisent directement. Cette page met en correspondance ce que vous connaissez déjà avec l'API Qora.

---

## Correspondance des concepts

| Concept TanStack Query | Équivalent Qora |
| ---------------------- | --------------- |
| `QueryClient` | `QoraClient` |
| `QueryClientProvider` | `QoraScope` |
| `useQuery` | `QoraBuilder<T>` (widget) ou `useQuery<T>` (hook) |
| `useMutation` | `QoraMutationBuilder` (widget) ou `useMutation` (hook) |
| `useInfiniteQuery` | `InfiniteQueryBuilder` (widget) ou `useInfiniteQuery` (hook) |
| `useQueryClient` | `context.qora` / `useQueryClient()` (hook) |
| `useIsFetching` | `useIsFetching()` (hook) ou `client.fetchingCountStream` |
| `useIsMutating` | `useIsMutating()` (hook) ou `client.activeMutations` |
| `QueryObserver` | `client.watchQuery<T>` / `InfiniteQueryObserver` |
| `persistQueryClient` | `PersistQoraClient` |
| `DevTools` | `qora_devtools_overlay` (in-app) / extension IDE |

## Référence API : requêtes

### Options

| Option TanStack Query | Équivalent Qora | Notes |
| --------------------- | --------------- | ----- |
| `queryKey` | `queryKey` | Même concept. TanStack utilise des tableaux ; Qora utilise `List<dynamic>`. Égalité structurelle dans les deux. |
| `queryFn` | `fetcher` | Même rôle. Renommé. |
| `enabled` | `enabled` | Identique. |
| `staleTime` | `staleTime` | Identique. `Duration.zero` = toujours périmé. |
| `gcTime` (`cacheTime` en v4) | `cacheTime` | Qora utilise le nom de la v4. |
| `retry` | `retryCount` | Renommé. Valeur par défaut `3` dans les deux. |
| `retryDelay` | `retryDelay` / `retryDelayCalculator` | Backoff exponentiel par défaut. `retryDelayCalculator` accepte une fonction recevant l'index de tentative. |
| `refetchOnWindowFocus` | `refetchOnWindowFocus` | Identique. Nécessite `FlutterLifecycleManager`. |
| `refetchOnReconnect` | `refetchOnReconnect` | Identique. Nécessite `FlutterConnectivityManager`. |
| `refetchOnMount` | `refetchOnMount` | Sémantique identique. `null` hérite de `QoraClientConfig.refetchOnMount`. |
| `refetchInterval` | `refetchInterval` | Identique. Actif tant qu'au moins un abonné est présent. |
| `networkMode` | `networkMode` | Même nom, mêmes trois valeurs : `online`, `always`, `offlineFirst`. |
| `select` | `state.map()` / `state.mapSuccess()` | Pas d'option ; utiliser la méthode `map` sur le `QoraState<T>` retourné. |
| `placeholderData: keepPreviousData` | Intégré dans `Loading` / `Failure` | Pas d'option nécessaire. `Loading<T>` porte toujours `previousData: T?`. Le comportement est toujours actif. |
| `initialData` | `client.hydrateQuery<T>(key, data)` | Pré-alimente le cache avant le premier abonnement. |
| `initialDataUpdatedAt` | `client.hydrateQuery<T>(key, data, updatedAt: ...)` | Passer `updatedAt` pour contrôler l'évaluation de péremption. |
| `structuralSharing` | Non disponible | Dart ne fait pas de partage structurel par défaut. |
| `notifyOnChangeProps` | Non disponible | Toutes les transitions d'état émettent sur le stream. |

### Valeurs de retour

| Champ TanStack Query | Équivalent Qora |
| -------------------- | --------------- |
| `status: 'pending'` | `state is Initial` ou `state is Loading(previousData: null)` |
| `status: 'error'` | `state is Failure` |
| `status: 'success'` | `state is Success` |
| `fetchStatus: 'fetching'` | `fetchStatus == FetchStatus.fetching` |
| `fetchStatus: 'paused'` | `fetchStatus == FetchStatus.paused` |
| `fetchStatus: 'idle'` | `fetchStatus == FetchStatus.idle` |
| `isLoading` | `state.isLoading` |
| `isFetching` | `fetchStatus == FetchStatus.fetching` |
| `isError` | `state.isError` |
| `isSuccess` | `state.isSuccess` |
| `isPending` | `state is Initial \|\| state is Loading(previousData: null)` |
| `data` | `state.dataOrNull` ou `Success(:final data)` via correspondance de motifs |
| `error` | `state.errorOrNull` ou `Failure(:final error)` via correspondance de motifs |
| `dataUpdatedAt` | `state.updatedAt` (extension, non-null uniquement sur `Success`) |
| `errorUpdatedAt` | Non disponible comme champ séparé |
| `isFetchedAfterMount` | Non disponible |

## Référence API : mutations

### Options

| Option TanStack Query | Équivalent Qora |
| --------------------- | --------------- |
| `mutationFn` | `mutator` — renommé. |
| `onMutate` | `MutationOptions.onMutate` — même rôle. La valeur de retour devient `TContext`. |
| `onSuccess` | `MutationOptions.onSuccess` — même rôle. Signature : `(TData, TVariables, TContext?)`. |
| `onError` | `MutationOptions.onError` — même rôle. Signature : `(Object, TVariables, TContext?)`. |
| `onSettled` | `MutationOptions.onSettled` — même rôle. |
| `retry` | `MutationOptions.retryCount` — renommé. Valeur par défaut `0` (pas de retry). |
| `retryDelay` | `MutationOptions.retryDelay` — backoff exponentiel. |
| `networkMode` | Non par mutation dans Qora — la mise en file hors-ligne est contrôlée par `MutationOptions.offlineQueue: true`. |
| `variables` | Paramètre de type `TVariables` — même concept. |
| `context` (snapshot de rollback) | Paramètre de type `TContext` — même concept. Retourné par `onMutate`, transmis à `onError`/`onSettled`. |

### Valeurs de retour / state

| Champ TanStack Query | Équivalent Qora |
| -------------------- | --------------- |
| `status: 'idle'` | `state is MutationIdle` |
| `status: 'pending'` | `state is MutationPending` |
| `status: 'error'` | `state is MutationFailure` |
| `status: 'success'` | `state is MutationSuccess` |
| `isIdle` | `state.isIdle` |
| `isPending` | `state.isPending` |
| `isError` | `state.isError` |
| `isSuccess` | `state.isSuccess` |
| `data` | `state.dataOrNull` |
| `error` | `state.errorOrNull` |
| `variables` | `state.variablesOrNull` |
| `mutate(variables)` | `mutate(variables)` — fire-and-forget, erreurs capturées dans le state |
| `mutateAsync(variables)` | `mutateAsync(variables)` — awaitable, erreurs propagées |
| `reset()` | `reset()` — transitions vers `MutationIdle` |

## Référence API : `QueryClient` vers `QoraClient`

| Méthode TanStack Query | Équivalent Qora |
| ---------------------- | --------------- |
| `invalidateQueries({ queryKey })` | `client.invalidate(key)` |
| `invalidateQueries(filter)` | `client.invalidateWhere(predicate)` ou `client.invalidateQueries(filter)` |
| `setQueryData(queryKey, updater)` | `client.setQueryData<T>(key, data)` |
| `getQueryData(queryKey)` | `client.getQueryData<T>(key)` — retourne `T?` |
| `getQueryState(queryKey)` | `client.getQueryState<T>(key)` — retourne `QoraState<T>` |
| `prefetchQuery` | `client.prefetch<T>(key, fetcher, options)` |
| `resetQueries` | `client.removeQuery(key)` ou `client.clear()` |
| `cancelQueries` | Non disponible dans Qora v1 |
| `isFetching(queryKey)` | `client.isFetchingCount` (compteur global) |
| `setDefaultOptions` | `QoraClientConfig.defaultOptions` (défini à la construction) |
| `getDefaultOptions` | `client.config.defaultOptions` |
| `mount` / `unmount` | Géré par `QoraScope` / cycle de vie des widgets |
| `clear()` | `client.clear()` |

## Différences clés

### Les classes scellées remplacent les chaînes de statut

TanStack Query retourne des objets simples avec des champs `status` sous forme de chaîne et des raccourcis booléens (`isLoading`, `isError`, etc.). Qora retourne une classe scellée `QoraState<T>`. La correspondance de motifs exhaustive de Dart 3 est appliquée à la compilation, éliminant les branches d'état non gérées :

```dart [user_screen.dart]
// TanStack Query (React)
if (isLoading) return <Spinner />;
if (isError) return <Error error={error} />;
return <UserCard user={data} />;

// Qora (Dart 3)
return switch (state) {
  Initial() || Loading(previousData: null) => const CircularProgressIndicator(),
  Failure(:final error, previousData: null) => ErrorScreen(error),
  _ => UserCard(state.dataOrNull!),
};
```

Le bras `_` couvre `Success`, `Loading(previousData != null)` et `Failure(previousData != null)` — tous les cas où les données sont disponibles.

---

### `previousData` est toujours présent, pas opt-in

TanStack Query v4 avait `keepPreviousData: true`. TanStack Query v5 l'a remplacé par `placeholderData: keepPreviousData`. Dans Qora, `Loading<T>` et `Failure<T>` portent toujours `previousData: T?`. Il n'y a pas d'option pour désactiver ce comportement. Les données périmées sont toujours accessibles pendant les rafraîchissements en arrière-plan et après les erreurs légères.

```dart [user_screen.dart]
// Pattern : afficher les données périmées avec un indicateur de progression pendant un rafraîchissement en arrière-plan
case Loading(:final previousData) when previousData != null:
  return Stack(children: [
    UserCard(previousData),
    const LinearProgressIndicator(minHeight: 2),
  ]);
```

---

### `select` est une méthode, pas une option

TanStack Query dispose d'une option `select` pour transformer les données avant de les retourner au composant. Qora expose les méthodes `map` et `mapSuccess` sur `QoraState<T>` :

```dart [main.dart]
// TanStack Query
useQuery({ queryKey, queryFn, select: (data) => data.name });

// Qora
final nameState = userState.map((user) => user.name);
// QoraState<User> → QoraState<String>
// previousData dans Loading/Failure est également transformé
```

---

### `gcTime` s'appelle `cacheTime`

TanStack Query v5 a renommé `cacheTime` en `gcTime`. Qora utilise `cacheTime` (le nom de la v4). La sémantique est identique : délai avant qu'une entrée inactive soit collectée par le garbage collector après l'annulation de son dernier abonné.

---

### API Widget vs API Hook

L'API principale de React Query est basée sur les hooks (`useQuery`, `useMutation`). Qora fournit les deux :

- `QoraBuilder<T>` et `QoraMutationBuilder` fonctionnent dans tout `StatelessWidget` ou `StatefulWidget`.
- `useQuery<T>` et `useMutation` nécessitent `HookWidget` de `flutter_hooks`.

Les deux partagent le même cache `QoraClient`. Si un `QoraBuilder` a déjà récupéré `['users', id]`, `useQuery` l'obtient au premier frame depuis la même entrée.

---

### `QoraScope` est un `StatefulWidget`, pas un `Provider`

TanStack Query utilise `QueryClientProvider` (contexte React). Qora utilise `QoraScope`, un `StatefulWidget` qui expose le client à son sous-arbre via un `InheritedWidget` interne. Accéder au client depuis n'importe quel descendant via `context.qora` ou `QoraScope.of(context)`.

---

### La persistance nécessite un enregistrement explicite des sérialiseurs

Les plugins de persistance de TanStack Query (ex. `createSyncStoragePersister`) utilisent `JSON.stringify` / `JSON.parse` sur le cache de requêtes complet. Qora nécessite un enregistrement explicite des sérialiseurs par type car Dart n'a pas de réflexion à l'exécution :

```dart [main.dart]
// Requis avant hydrate()
client.registerSerializer<User>(
  QoraSerializer(toJson: (u) => u.toJson(), fromJson: User.fromJson),
  name: 'User',
);
await client.hydrate();
```

La chaîne `name` doit être stable entre les versions de l'application et les modes de build (y compris les builds obfusqués).

---

### Pas d'équivalent React Suspense

TanStack Query supporte `suspense: true` pour s'intégrer avec React Suspense. Flutter n'a pas de mécanisme équivalent. Tout état asynchrone est géré via la correspondance de motifs sur `QoraState<T>` dans l'arbre de widgets.

## DevTools

Les DevTools de TanStack Query sont un panneau navigateur installé comme composant React. Qora fournit deux surfaces :

- **Overlay in-app** : installer `qora_devtools_overlay` sous `dev_dependencies`, encapsuler l'application avec `QoraInspector`. Absent des builds de production. Disponible aujourd'hui.
- **Extension IDE** : installe un onglet dédié dans Flutter DevTools. En développement actif, pas encore publié.

Voir [Vue d'ensemble des DevTools](/devtools/overview) pour les instructions de configuration.
