**Qora** est une bibliothèque de gestion du state serveur pour Dart et Flutter, modélisée sur l'architecture de [TanStack Query](https://tanstack.com/query).

## Problématique

Le state serveur est catégoriquement différent du state UI local. Il réside sur un système distant, possède un cycle de vie indépendant et nécessite une synchronisation explicite avec le client. Qora adresse cette classe de problèmes :

- **Mettre en cache** les données avec un TTL configurable et les servir instantanément lors des requêtes suivantes
- **Revalider** les données périmées en arrière-plan sans bloquer l'UI
- **Dédupliquer** les requêtes concurrentes pour la même clé en un seul appel réseau
- **Réessayer** automatiquement les requêtes échouées avec un backoff exponentiel
- **Invalider** les entrées de cache ciblées après les mutations, déclenchant des refetchs réactifs

## API principale

### Récupération ponctuelle

```dart [user_repository.dart]
final user = await client.fetchQuery<User>(
  key: ['users', userId],
  fetcher: () => api.getUser(userId),
  options: const QoraOptions(staleTime: Duration(minutes: 5)),
);
```

Comportement du cache : les données fraîches sont retournées immédiatement sans appel réseau. Les données périmées sont retournées immédiatement pendant qu'un refetch en arrière-plan est déclenché.

### Stream réactif

```dart [user_repository.dart]
client.watchQuery<User>(
  key: ['users', userId],
  fetcher: () => api.getUser(userId),
).listen((state) {
  switch (state) {
    case Success(:final data):  render(data);
    case Failure(:final error): showError(error);
    default: {}
  }
});
```

### Invalidation après mutation

```dart [post_repository.dart]
await api.createPost(payload);

// Invalider une clé exacte
client.invalidate(['posts', postId]);

// Invalider par prédicat
client.invalidateWhere((key) => key.firstOrNull == 'posts');
```

### Mises à jour optimistes

```dart [user_repository.dart]
final snapshot = client.getQueryData<User>(['users', userId]);
client.setQueryData(['users', userId], user.copyWith(name: newName));

try {
  await api.updateUser(userId, newName);
} catch (_) {
  client.restoreQueryData(['users', userId], snapshot);
}
```

## Architecture

Qora est organisé en trois packages complémentaires.

### `qora` : noyau Dart pur

Aucune dépendance Flutter. Fonctionne dans tout environnement Dart :

- Outils CLI et scripts
- Services backend Dart
- Tests unitaires sans le runner Flutter
- Packages de logique métier partagée

### `qora_flutter` : intégration Flutter

Une couche de liaison de widgets sur `qora` :

- `QoraScope` : `InheritedWidget` qui fournit le client à l'arbre de widgets
- `QoraBuilder<T>` : récupère et reconstruit à chaque transition d'état ; le builder reçoit `(BuildContext, QoraState<T>, FetchStatus)`
- `QoraStateBuilder<T>` : observe l'état sans déclencher de fetch
- `QoraMutationBuilder<TData, TVariables, TContext>` : machine d'état de mutation avec callbacks de cycle de vie
- `InfiniteQueryBuilder<TData, TPageParam>` : flux paginés avec `fetchNextPage` / `fetchPreviousPage`
- `NetworkStatusIndicator` / `NetworkStatusBuilder` : bannière hors-ligne et widgets adaptés à la connectivité
- `FlutterLifecycleManager` : invalide les requêtes quand l'application revient au premier plan
- `FlutterConnectivityManager` : invalide les requêtes quand la connexion réseau est rétablie
- `context.qora` : raccourci `BuildContextExtension` pour `QoraScope.of(context)`

### `qora_hooks` : intégration flutter_hooks

API basée sur les hooks pour les projets utilisant [flutter_hooks](https://pub.dev/packages/flutter_hooks) :

- `useQuery<T>` : équivalent à `QoraBuilder` sans le callback de builder imbriqué
- `useMutation<TData, TVariables>` : retourne un `MutationHandle` avec `mutate`, `isPending`, `isError`
- `useInfiniteQuery<TData, TPageParam>` : retourne un `InfiniteQueryHandle` avec `fetchNextPage`
- `useQueryClient` : retourne le `QoraClient` le plus proche pour les opérations impératives
- `useIsFetching` / `useIsMutating` : indicateurs `bool` globaux ; reconstruction uniquement au passage par zéro

### DevTools

- **Overlay in-app** *(stable)* : panneau draggable affichant l'état du cache en direct sans quitter le simulateur
- **Extension IDE** *(en développement)* : inspecteur de requêtes, timeline des mutations et visualiseur de diff du cache dans Flutter DevTools — pas encore publié
- Les deux surfaces exposent les mêmes événements : transitions d'état, durées de fetch, diffs optimistes et annulations

## Principes de conception

### Gestion déclarative des états

Le consommateur déclare ce qu'il faut afficher pour chaque variante d'état. Qora gère la planification des fetchs, la résolution du cache et la revalidation.

### Sûreté du typage par construction

`QoraState<T>` est une classe scellée. Le compilateur Dart impose une correspondance de motifs exhaustive, éliminant les branches d'état non gérées à la compilation.

### Valeurs par défaut configurables

La configuration par défaut (`staleTime: Duration.zero`, `cacheTime: 5 minutes`, `retryCount: 3`, backoff exponentiel) est remplaçable globalement via `QoraClientConfig` ou par requête via `QoraOptions`.

## Comparaison des fonctionnalités

Le tableau ci-dessous compare Qora aux bibliothèques de la même catégorie : `cached_query` et `flutter_query`. BLoC et Provider ciblent le state client (logique UI, formulaires, navigation) et sont hors du périmètre de cette comparaison.

| Fonctionnalité                                   | **Qora**            | cached\_query  | flutter\_query |
| ------------------------------------------------ | ------------------- | -------------- | -------------- |
| Stale-While-Revalidate                           | Intégré             | Intégré        | Intégré        |
| Déduplication des requêtes                       | Intégré             | Intégré        | Intégré        |
| Mises à jour optimistes + rollback               | Intégré             | Manuel         | Intégré        |
| Retry automatique (backoff exponentiel)          | Intégré             | Intégré        | Intégré        |
| Requêtes infinies / paginées                     | Intégré             | Intégré        | Intégré        |
| Requêtes dépendantes (`dependsOn`)               | Intégré             | Manuel         | ❌             |
| Cancel tokens                                    | Intégré             | ❌             | ❌             |
| Persistance hors-ligne (Hive, Isar...)           | Intégré             | Plugin         | ❌             |
| **File de mutations hors-ligne**                 | **Intégré**         | ❌             | ❌             |
| **Refetch adapté à la connectivité**             | **Intégré**         | Basique        | ❌             |
| **DevTools**                                     | **Intégré**         | Plugin         | ❌             |
| API Hooks (useQuery, useMutation...)             | Plugin (qora_hooks) | ❌             | Intégré        |
| Dart pur (pas de dépendance Flutter)             | ✅                  | ❌             | ❌             |

**Légende :** `Intégré` : zéro configuration, inclus dans le package · `Plugin` : disponible via un package séparé · `Manuel` : réalisable mais nécessite une implémentation personnalisée · `❌` : non supporté.

::note
**Le state serveur et le state client sont des domaines de problèmes distincts.**

**Le state serveur** est la donnée qui provient d'un serveur distant. Il possède un cycle de vie indépendant du client : il est récupéré, mis en cache, devient périmé et doit être re-synchronisé. Les préoccupations techniques principales sont le cache, la déduplication, les retries et la résilience hors-ligne.

**Le state client** est la donnée qui existe exclusivement dans le processus applicatif : valeurs de formulaires, onglet sélectionné, état d'animation, tokens de session en mémoire. Il n'a pas de cycle de vie réseau et ne devient pas périmé.

Les applications en production nécessitent les deux couches. Qora gère le state serveur ; BLoC, Riverpod ou Provider gèrent le state client. Les deux domaines sont complémentaires et répondent à des besoins différents.
::

## Cas d'utilisation cibles

- **Applications CRUD** : utilisateurs, produits, commandes avec écrans de liste et de détail réactifs
- **Flux sociaux** : publications, likes et commentaires synchronisés sur plusieurs écrans
- **Tableaux de bord** : polling de données en direct avec revalidation en arrière-plan
- **E-commerce** : flux de panier et de paiement avec UI optimiste
